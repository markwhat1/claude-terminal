## Section 1: ADHD design principles, applied concretely

The recon converges on one rule restated by every E-lane and by Mark's own J.O.T. model: surface ONE recommended next thing as the hero, turn the volume down on everything else, and make the finish pay out. This section binds each principle to a concrete, buildable behavior. It is gradable against metric 1.

A board that only demands becomes a guilt object and gets avoided (E5 P2). So the finish must pay out. But a board that piles five interactive behaviors on the user at first run is its own first-run overwhelm. The resolution: ONE dedicated carrot adds a resting INTERACTION affordance (the done-lane payoff, 1.5); the remaining motivational weight is carried by the calm visual layer and the goal-gradient (1.10), which add no new interaction surface. This is restraint in interaction count, not a claim that the board has only one motivating element; the board ships seven calm behaviors and says so (Honesty about scale, above). The escape valve (re-roll, 1.6) and capture (1.3) move to Phase 2. The coaching rituals (stall-interrupt 1.8, lock-in 1.9) move to Phase 3, default OFF. The cue all of this reuses is APP-OPEN itself (E7 P5, the meta-analysis finding that cue-bound implementation intentions beat willpower): Phase 1's Home-on-open is the unstyled precursor of the Phase-3 morning intake, the same cue with lock-in attached, not a second mechanism.

### 1.1 Hero = one thing (a hard affordance budget)

- The Home view has a single dominant hero card rendering exactly ONE item. In Phase 1 it is the producer-ordered needs-you head with a single-field override so a deadline or an almost-done item beats newest-commit (1.11); from Phase 2 it is `rankItems(...)[0]` (Section 5). Never a row of equal cards, never the session grid.
- Dominance has exactly three levels: dominant (hero card), sub-dominant (the rest of the needs-you list as compact rows), subordinate (the live-session strip). A fourth level destroys the contrast and is banned in the component spec (Section 6).
- The hero carries a HARD affordance budget (6.3): exactly one full-weight primary action button, one quiet "not now", and one quiet copy. No more. Re-roll (Phase 2) is the "not now"; lock-in (Phase 3) does NOT add a second persistent button on the resting hero, it lives at first-open intake only (1.9). A menu of equal opposites on one card (re-roll = defer, lock-in = commit) is the choice-overload the one-thing hero exists to remove (E7 C7, E1 AP4), so the two are never both persistent controls at once.
- The Phase-1 "not now" slot is INTENTIONALLY EMPTY (decided, not unfilled): re-roll is Phase 2, so in Phase 0/1 the hero renders no "not now" control and the slot stays reserved. The risk this leaves is a deterministic single-field override (1.11) that re-pins the same dread item as hero every poll with no calm exit. Phase 1 constrains the override to bound that: it elevates a card to hero ONLY when the override actually changes board order (a time-sensitive or dodAlmost card jumping ahead of the producer head). When the override would pick the same card the producer head already is, the hero is just the producer head, which board order already varies poll-to-poll (`git.age_days` shifts as commits land), so no card is permanently no-exit-pinned before re-roll ships. This is latent on today's data (no avoidance item is the live hero, 1.11) and stated so the calmest day-one board does not silently become a no-exit guilt billboard (E5 5.1, E1 P8). M8a asserts the override only fires when it reorders.
- Keyboard floor (Phase 1, asserted, and credited as an ACTIVATION-ENERGY guarantee not an a11y checkbox): the Home REGION (the hero card container) is focused on mount, NOT the primary action button. Auto-focusing the loudest `bg-attention` pixel on every mount (startup, last-tab-close routing to Home, the entry affordance, `startupView:'home'`) reads as "the app wants me to do this NOW", a soft demand on the calm landing, the over-saturation the capped hero (1.4) and heat-signal-once (6.3) rules remove elsewhere. So the resting state is a calm board with no focus ring on a demand. The hero primary button (or the Copy-only fallback, 6.3) is the FIRST TAB STOP in the Home region and Enter-activates, so mouse-free reach is Tab+Enter, two keystrokes, still cheap and still a real activation-energy saving over a mouse hunt. Copy is the next Tab stop; the strip rows and the "+N more"/"N paused" disclosures are focusable and Enter-jump/Enter-expand. No custom arrow-key model in Phase 1 (shadcn `Button` is focusable by default). M8a asserts: the Home region (not the button) is the active element on mount, the hero primary is the first focusable element after the region, and it activates on Enter (6.3). The headline action navigates AWAY from Home to the spawning tab (1.5b names that tradeoff); the calm landing is preserved by not yanking focus to the demand on arrival.
- Home-active global chords (re-grounded against the real registry, verified at HEAD `ce2e9e0`): landing on Home means `activeTabId === HOME_TAB_ID`, which is in no real tab. `keybindings.ts` is a DECLARATIVE `Keybinding[]` registry (`{mod,key,action(ctx)}`, `:53-67`) dispatched through a KeybindingContext built in `App.tsx`; there are no editable per-line chord seams at `:40/:58/:59`. Two of the three previously-cited "crashes" are already null-guarded today: Ctrl+F4 (`:59`) is `const id = ctx.activeTabId(); if (id) ctx.closeTab(id);` and `HOME_TAB_ID` is truthy so it would call `closeTab` on a non-tab (the one REAL behavior bug, harmless-but-wrong); Ctrl+\` (`:58`) is `ctx.newDefaultShellTab(ctx.activeTabId() ?? undefined)` and `getTab(HOME_TAB_ID)` returns undefined so it harmlessly falls back to the default cwd. The genuine misbehavior is `cycleTab` (`:37-43`): `findIndex` returns -1 with Home active so `next = (-1+1+len)%len = 0`, which jumps to `tabs[0]` (this is actually the desired "cycle into the real tabs" behavior, so nothing to change there). M3a-iii hardens the KeybindingContext CLOSURES in `App.tsx` (the dispatcher level, NOT edits to the registry array): the Ctrl+F4 closure no-ops when `activeTabId === HOME_TAB_ID`; the Ctrl+\` closure passes `undefined` not the Home id (it already does, via the `getTab` undefined fallback, so this is a tidy-up + a pinning test); Ctrl+Tab's cycle-into-tabs-0 behavior is asserted as correct, not changed. The round-5 "harden the StatusBar active-tab lookup at `App.tsx:585`" target was wrong (StatusBar takes a `tabs` array and never does `tabs.find(activeTabId)`, verified) and the `window:setTitle` deref claim is dropped as unfounded. Tests assert each chord with Home active does not crash and behaves as specified.
- The 90%-killer is hero-eligible and is driven OFF THE PRODUCER, never re-derived (Section 4.4, Section 5.3 Tier 3).

### 1.2 Calm by default

There is no second `--attention` surface on Home: the earlier "single highest needs-you dot" accent is DROPPED (it appeared once, had no class binding and no test, silently widened the one-accent budget to two, and contradicted the 6.2 sub-dominant row spec, which carries no per-row accent). The needs-you rows carry their verbatim `ageColorClass` per 4.3 (the age bands), which is the heat signal; they do not get a second saturated dot competing with the hero button. Per-session identity in the strip is a thin desaturated tint from `--project-hue`, applied as a left border via the house arbitrary-value class pattern already used across the app (`border-l-2 border-[hsl(var(--project-hue)_30%_35%)]`, mirroring `StatusBar.tsx:37`, `Tab.tsx:89`, `TabBar.tsx:100`), never a fill. The round-2 "never inline style for the hue" advice is dropped: the codebase already sets `--project-hue` inline (`App.tsx:109`, `ProjectSidebar.tsx:69-70`) and reads it via `hsl` arbitrary classes; match that pattern rather than fight it. M8a asserts exactly one `bg-attention` element renders on Home.
- No uninvited motion. On a state re-read or a tab-status event, the renderer updates in place. No auto-scroll, no auto-reorder of the visible list, no auto-expand, no carousels. The list order is recomputed but the DOM diff is keyed by stable `id` so React updates content without reflow churn. Honor `prefers-reduced-motion`: every animation is gated behind a `motion-safe:` Tailwind variant.
- The two existing animated glyphs are `animate-spin` on the `working` glyph (`TabIndicator.tsx:15`, Loader2) and `animate-pulse` on the `requires_response` glyph (`TabIndicator.tsx:27`, MessageCircle). Both are BARE today (no `motion-safe:` gate). Because the strip REUSES `TabIndicator` (6.4), the reduced-motion promise is silently broken unless the source is gated. M2 is a standalone one-change/one-test milestone that adds `motion-safe:` to both lines; it also fixes the existing tab bar.
- Relative time, never raw timestamps. Every temporal value renders through one `formatRelative` helper. The working row shows count-UP elapsed since `statusSince` ("working 12m"); the idle row shows idle DURATION ("idle 6m"). These are two different computations off two different anchors and the M9 test asserts they are distinct (6.4). Never a countdown.
- The active count-up is COARSENED to minute resolution on the ambient Home surface ("1m", "2m", not "1:03", "1:04"). A digit changing every second in the periphery is the looping-peripheral-motion anti-pattern (E4 A7) and a number racing upward on a stalled task manufactures the time-anxiety count-up was chosen to avoid (E3 P7). Recompute on the ~20s poll tick, not a `setInterval(1000)`. Under `prefers-reduced-motion`, coarsen further or freeze to the last value. True per-second ticking is reserved for an explicit user-started focus timer (Phase 3, E7 F7), never ambient Home.
- Whitespace carries hierarchy via the concrete grid in Section 6.1 and the concrete type/contrast/density classes in 6.2, not adjectives.

It clears when the tab first reaches `idle` (the injection lands).
- A CALM HONEST HEARTBEAT across the boot window (the time-blind-disengagement fix): the 3-8s CLI boot otherwise has NO temporal cue, so a static line at second 7 is indistinguishable from a hung line, trading "reads as working" for "reads as nothing-happening" (the E3 now-vs-not-now trigger 1.5b exists to prevent). This is a coarse, honest, NON-metronome cue: one step at one threshold, never a per-second ticker (which would be the looping-peripheral-motion anti-pattern), so the time-blind brain reads "still going, not hung" and does not disengage at second 6. The testable property that actually matters: the pending injection + timeout are armed in MAIN BEFORE the handler resolves, so a renderer reload after the await cannot orphan the query. The OUT-OF-APP CLI boot is bounded by the 30s fail-safe and owned by this pending UI; the rest is visibly in progress, never blank.

- SEPARATE THE WAITING CLOCK FROM THE DISPLAY CLOCK : `statusSince` drives the display string ("idle 6m"), but an `idle -> requires_response` transition is a real status change, so it RESETS `statusSince` even though the human has been ignoring the tab since the earlier idle. The idle-age floor (5.2) and the Tier-2 longer-idle tie-break (5.4) must therefore read a DEDICATED `waitingSince` anchor, set when the tab first becomes human-waiting and NOT reset by an `idle -> requires_response` transition within a continuous waiting span. `waitingSince` is set on entering a human-waiting status (`idle` with `firstActivityAt !== null`, or `requires_response`) only if it is currently null; it clears to null on entering `working` (a new turn started, the human is no longer waiting). So the escalation heuristic ages the tab from when the human FIRST had to act, not from the latest notification:
 ```ts
 updateStatus(id: string, status: TabStatus): void {
 const tab = this.tabs.get(id);
 if (!tab) return;
 const now = Date.now;
 if (tab.status !== status) tab.statusSince = now;
 if (status === 'working') {
 if (tab.firstActivityAt === null) tab.firstActivityAt = now;
 tab.waitingSince = null; // new turn: human no longer waiting
 } else if ((status === 'idle' && tab.firstActivityAt !== null) || status === 'requires_response') {
 if (tab.waitingSince === null) tab.waitingSince = now; // span start, not reset by overlay
 }
 tab.lastActivityAt = now;
 tab.status = status;
 }
 ```

### 3.1 Open Claude with query (write-after-ready)

The cwd is RESOLVED FROM THE HERO'S PROGRAM, not the active project. The hero is a program-board card whose `repos[0]` is usually NOT the active project, so creating the tab in `activeProjectIdRef.current` would run the canned query against the wrong tree (the same wrong-tree defect the plan disables for the remote path, silently present on the local path 3.6 treats as security-relevant).

HOW the cwd actually reaches the spawn : M10b adds an `explicitCwd` param to `tab:create` mirroring `tab:createShell`, so the action spawns in `WORKSPACE_ROOT + program.repos[0]` WITHOUT calling `project:add`. The contradiction is resolved: the install is required, the artifact is git-ignored.

1. Compose the query through `composeClaudeQuery` (the choke point, 3.4). It is the ONLY producer of the query string and returns a branded `ClaudeQueryLine` type (3.6) so a raw string cannot reach the injection write.
2. Resolve the target cwd from `program.repos[0]` via `resolveProgramProject` (above): an already-open project returns its id; an unopened repo routes through the `explicitCwd` param; an unresolvable dir falls back to Copy-only. The no-active-project case is NOT reachable in Phase 1 : Home renders only in `appState:'running'`, which `App.tsx:286-300` reaches only after `startSession()` resolves a project, and R-10 Option A auto-resumes a last-session directory, so `workDir` is always non-null when Home is live. Forcing the path anyway would THROW: `tab:create` guards `if (!workDir) throw 'Session not started'` at `ipc-handlers.ts:365` BEFORE `cwd` is computed, and `tab:createShell` has the same guard at `:529` before `cwd = explicitCwd || workDir` at `:531`, so `explicitCwd` does NOT bypass the no-workDir throw. If a future phase ever needs it, the fix is to reorder the guard to `const cwd = explicitCwd ?? workDir; if (!cwd) throw` in both handlers so `explicitCwd` short-circuits, but Phase 1 does not need that change.
3. The renderer click handler calls the single MAIN `claude:injectQuery` handler and AWAITS it once. The earlier "same tick, no await, app's share is provably zero" framing was unbuildable. Instead, `claude:injectQuery` is a single MAIN handler that: creates the tab via the `explicitCwd` route, ARMS the pending-injection entry + the 30s timeout in MAIN, and RETURNS the new tab id. The renderer awaits that ONE round-trip, then calls `setActiveTabId(id)` and arms the 1.5b pending affordance on the spawning tab. The load-bearing testable property (replacing the slogan): the pending injection + timeout are armed in MAIN BEFORE the handler resolves, so a renderer reload after the await cannot orphan the query. The app's controllable latency is one IPC round-trip (1.13); the OUT-OF-APP CLI boot that follows is bounded by step 7's 30s fail-safe and surfaced by the pending state (1.5b), never blank.
4. The once-flag gate lives in MAIN and is pinned to the CONVERGENCE POINT, not one case (verified: a freshly spawned tab reaches its FIRST idle via the `tab:ready` SessionStart path, which calls `updateStatus(tabId,'idle')` at `hook-router.ts:104`; the `tab:status:idle` case at `:125-132` fires only LATER, on the Stop hook after the user's first turn, so gating only `:125` means the canned query never types in on click). This is a NET-NEW cross-module main feature, not the renderer sketch R3 floated (`HookRouterDeps` does NOT import `ptyManager` at `:6-16`, and main's idle observation only sets status + fires a notification; zero injection plumbing in main today). M10c's change-list names all pieces: (a) give the injection write access to `ptyManager.write` (extend `HookRouterDeps` or relocate the write to `index.ts`), changing the deps contract; (b) a `pendingInjection` Map + per-tab timer OWNED in MAIN; (c) the gate hooked at the `tab:updated` EMISSION (`hook-router.ts:167-170`) checking `updated.status === 'idle'` for a tracked tab (the point where BOTH the `tab:ready` first-idle and the later `tab:status:idle` converge), OR equivalently a check at BOTH `updateStatus(...,'idle')` sites (`:104` AND `:126`). When a tracked tab first reaches `idle` AND `!injected`, do the write and set `injected = true`. Every later `idle` (the `--resume` second SessionStart fire, a `/clear`, the normal idle-after-each-turn) is ignored. Because the intent + once-flag + timer live in MAIN, a renderer RELOAD during the multi-second CLI boot (`App.tsx:302-316`) cannot wipe the pending Map or cancel the timer. M10c's test MUST drive a REAL `tab:ready` message (`{sessionId, source:'startup'}`) as the first-idle case and assert exactly one `ptyManager.write` with a trailing `\r`; then assert the Stop `tab:status:idle` and the `--resume` double-`tab:ready` are idempotent (no second write), and `tab:closed` clears the timer. A test that only sends `tab:status:idle` is REJECTED: that case never carries the first idle, so it would pass while the bug ships.
4b. Toast suppression on the injected tab's POST-TURN idle (the toast fires at the right event, and the test drives that event, not the wrong one): the "Claude has finished working" toast fires only in the `tab:status:idle` case (`hook-router.ts:130`), i.e. the LATER Stop-hook idle after the injected query's turn finishes; the FIRST idle via `tab:ready` (`:104`) carries NO `notifyTabActivity` call. So suppressing-and-testing at the `tab:ready` first idle would pass trivially (that event never notifies regardless of fix) while the real post-turn toast still fires for a non-MAIN-active injected tab, giving false assurance. The idle toast is suppressed only when `!isActive`, and `isActive` reads MAIN's `tabManager.getActiveTabId` (`hook-router.ts:60`), while step 3 sets only the RENDERER's active id; for a renderer-only-Home app the two diverge, so the fresh tab is often not MAIN-active. Fix: the injection path drives MAIN's active id (the handler sets the tab MAIN-active when the inject intent arrives) AND sets a do-not-notify flag on the tracked tab that PERSISTS through at least the first post-injection Stop idle, so the post-turn toast is suppressed regardless of active state. M10c's test drives a `tab:status:idle` (Stop-hook idle) for the injected tab while it is NOT renderer-active and asserts `notifyTabActivity`/`Notification` is not called (because the injection path set the do-not-notify flag), not just that the `tab:ready` first idle is quiet.
5. The write uses CR, never CRLF (R3 §(c)): MAIN composes `const line = query.replace(/\r?\n/g, '\r') + '\r'` and writes it to the pty. xterm emits `\r` on Enter (`Terminal.tsx:148-150`); the in-app refresh action already pushes a raw control byte (`App.tsx:209-212`, `\x0c`). A trailing `\r\n` can register as two submissions in a ConPTY TUI.
6. Cleanup: on `tab:removed`, MAIN deletes the pending entry and clears the timer so a dead tab cannot hold a stale write and the Map cannot grow unbounded. The gate ALSO verifies the PTY is live at write time: `ptyManager.write` is `this.ptys.get(tabId)?.process.write(data)` (`pty-manager.ts:66-68`), a silent no-op if the PTY died in the narrow window between the idle signal and the write landing, and by then the gate has already cancelled the 30s timeout and set `injected=true`, so the write would vanish with nothing surfacing the failure. So the gate checks `hasPty(tabId)` before write and, if absent, emits a `claude:injectStatus` failure (the same surface as the 30s timeout) rather than assuming the write reached a live PTY.
7. Fail-safe is MANDATORY (R3 §(d)): write-after-ready depends on `on-session-start.js` firing, which depends on `hookInstaller.install(cwd)` having run for that cwd (`ipc-handlers.ts:385-388`). For a freshly-added project an install/spawn race or a global Claude config override means no `tab:ready` arrives, the idle gate never fires, and the query is silently dropped. For an ADHD user who clicked "decide this" and walked away, a silent drop is the worst failure. So M10c ships a 30s timeout in MAIN (not optional) that emits a "Session failed to start, query not sent" event to the renderer with a one-click retry that re-runs the same intent into the same tab. The timeout-fires-and-surfaces case AND a renderer-reload-mid-flight case (the pending write is NOT lost) are both in the M10c test.
8. Permission-mode coupling, with the MECHANISM specified (not just the requirement): the dashboard-spawned tab inherits `state.permissionMode` (`ipc-handlers.ts:390`, `PERMISSION_FLAGS[state.permissionMode]`), and there is NO per-call permission param today, so in plan mode the `PERMISSION_FLAGS.plan = ['--plan']` bug (9.1; the real flag is `--permission-mode plan`) can error the tab at startup so it NEVER reaches `idle`, and the injection silently waits out the full 30s on every click. R-4's old "default is bypassPermissions so the common path is unaffected" claim is wrong: the tab inherits the WORKSPACE mode, not the default. The mechanism: M10b/M10c ADD a `permissionModeOverride?: PermissionMode` param to `tab:create` (alongside `explicitCwd`) and apply it at `:390` as `PERMISSION_FLAGS[override ?? state.permissionMode]`. The injection spawn passes `bypassPermissions` (the canned queries are safe), so a plan-mode workspace produces `--dangerously-skip-permissions` for the injection tab, not `--plan`, and the idle gate is reachable. M10c tests that an injection spawn in a plan-mode workspace produces `--dangerously-skip-permissions`, not `--plan`. The 30s failed-start message also reads "session may have failed to start; check permission mode" so the always-timeout case is legible if the override is ever removed. The `--plan` bug itself is still filed (9.1) for the user-facing plan-mode tabs.

The injection channel (this DOES add one new local IPC channel, full AGENTS.md treatment): `claude:injectQuery` is an `ipcMain.handle` (request/response) carrying `{ explicitCwd, query: ClaudeQueryLine }` to MAIN and RETURNING the new tab id (step 3), so the renderer awaits once and never owns the create-then-arm sequence; the MAIN handler creates the tab, arms the pending entry + 30s timeout, and resolves. Status/failure flow back over a `claude:injectStatus` broadcast. The earlier "no new channel" claim assumed renderer-owned state; moving the fail-safe to MAIN is worth one channel because it removes the only silent-drop path. Both channels get handler + preload + `global.d.ts` + registration test + an explicit remote decision (disabled remotely, see below). The renderer no longer owns a `pendingInjection` ref.

Why gate on `idle` not raw `tab:ready`: even MAIN collapses every hook event into a status update path; the load-bearing observable is the `idle` status transition. MAIN sees it first (`hook-router.ts:104`), which is exactly why the gate belongs there.

Remote parity (CORRECTION of R5 §D, flagged because it overrides the cited authority, 2.11): R5 §D ruled this action works remotely with no new channel. That is WRONG, and the plan overrides it deliberately. The remote `tab:create` handler IGNORES the dashboard's resolved cwd: `ws-bridge.ts:249-254` discards `_projectId`/`_worktree` and sends a bare `{type:'tab:create'}`; `web-remote-server.ts:316-323` hardcodes `state.workspaceDir` and `createTab(cwd, null, 'claude')` (verified, the cwd-discard R5 §D missed). ### 3.4 The PHI-scrub choke point: composeClaudeQuery

Per R6 item 1. The query is the only action that can carry free text into a PTY write and logs. Enforce in ONE place.

- `composeClaudeQuery` is the SOLE producer of the query string and returns a `ClaudeQueryLine` branded type (3.6). Do NOT scrub inside `PtyManager.spawn` (too late) and do NOT rely on scrubbing only at the log call (the write path would still carry raw text).
- Conservative default (the shipped behavior, the real guarantee): canned templates with ZERO free-text interpolation. `composeClaudeQuery({ action: KnownActionId, programSlug, programName, kind })` where `action` is a fixed set of canned prompts: `reviewTodos` ("review the open TODOs in this repo"), `summarizeChanges` ("summarize what changed on this branch"), `draftFirstVersion` ("Draft the first version of <programName> so I can review and send it", slot filled from the program NAME + fixed kind label only, never `repos[0]`, 1.7). A default that injects only a fixed verb + a producer-computed identifier cannot leak PHI even if a scrubber has a gap.
- "Zero interpolated free text reaches the composed body for any `KnownActionId`" is a NAMED, LOAD-BEARING regression test, not a convention enforced by review. The Phase-2/3 free-text opt-in cannot ship without explicitly deleting or modifying that test, so the policy is a gate whose removal is reviewable in the diff. The same applies to the `copy.text` whitelist (3.3).
- CONTENT VALIDATION on slug/name (a Phase-1 tested control, not policy-only): the slug/name leak-freedom is CONDITIONAL on `program.slug`/`program.name` being non-PHI dev identifiers, and they come from per-program YAMLs via the producer (an UNTRUSTED read source per 3.6), so a program named after a patient case would flow verbatim into `composeClaudeQuery`, the PTY write, the new tab's Haiku auto-name (500 chars, R-14), and the remote-broadcast `tab.name`. The plan validates the state.json PATH (`isStateJsonPathSafe`) but, asymmetrically, gave this input only a policy-only "the precondition is STATED" mitigation in a different repo this plan cannot enforce or test. So Phase 1 adds a tested `isSafeProgramIdentifier(slug|name)` pure predicate in `src/shared/` (mirroring `isStateJsonPathSafe`) that bounds length and rejects PHI shapes (long digit runs and the `scrubFreeText` patterns) BEFORE slug/name reach `composeClaudeQuery` or the spawned-tab name; on a hit the action falls back to slug-only or Copy-only. The producer-side contract (program names must be dev identifiers) stays the durable UPSTREAM fix, recommended to the program-board repo, but it is no longer the ONLY control. R-15 carries the residual. If Mark declines the predicate, R-15 must state plainly that Phase 1 ships with NO content control on this input (accepted risk).
- HONESTY about reach and specificity (1.7): the canned default is SAFE BUT GENERIC. For the live `incomplete-notes` card it composes a near-meaningless query because the specific deliverable lives in `dod.gaps[0]` (the forbidden slot). The plan bills it as safe-and-generic, never as both safe and highest-specificity. A Phase-2/3 opt-in MAY allow only the producer-COMPUTED `dod.gaps[0]` DoD-label text into the slot behind the scrubber as a tested exception; that is a documented tradeoff, not a Phase-1 default.
- Honesty about the choke point's reach (3.6): `composeClaudeQuery` constrains the dashboard action's CALL SITE only, not the `pty:write` CHANNEL. Every normal keystroke calls `writeToPty(tabId, rawString)` (preload `:50-51`) and the remote bridge sends raw bytes (`ws-bridge.ts:283-285`, `web-remote-server.ts:272-276`); the brand only stops a future `writeToPty(tab.id, detail)` at the dashboard injection call site, so a reviewer must not read the choke point as a channel-level control. The raw `pty:write` passthrough is ALSO remote-enabled: an authed remote client can `tab:create` a Claude session at the workspace root then `pty:write` arbitrary bytes into it or any tab, bypassing `composeClaudeQuery` entirely. The dashboard's injection-safety guarantee is therefore LOCAL-ONLY; the remote token is the only guard on the channel. Named as a residual risk in Section 10 (R-11), not buried as a 3.4 aside. The local renderer's own future code is the other unguarded path: a HomeView affordance calling `writeToPty(tabId, item.detail)` directly bypasses the brand; M0b's mapper assertion is extended to assert HomeView never passes `detail`/`blocked_on`/`dod.gaps` to `writeToPty` (3.6).
- The choke point is NOT the only free-text-to-LLM seam: the tab-namer (`tab-namer.ts:75`) ships `prompt.substring(0,500)` to Haiku for every auto-named tab, and the dashboard spawns a fresh tab per injection. Any free-text-bearing query opt-in (the Phase-2/3 `dod.gaps[0]` exception) MUST also gate the tab-namer (suppress auto-naming for dashboard-injected tabs, or run the namer prompt through `scrubFreeText`) so specificity does not leak to Haiku unscrubbed. The code gate ships with the opt-in; the residual risk is R-14 now.
- `scrubFreeText` is HARM-REDUCTION, not a control. It has no enabled Phase-1 caller (the shipped default has zero free text), but it is a PURE function and ships WITH A UNIT TEST beside its caller (M0b's prompt-prefix redaction, see below; else M0c) so the opt-in path, if ever enabled, inherits a tested scrubber rather than dead code. It CANNOT redact the primary PHI for this practice (patient NAMES, which a regex + name list cannot enumerate) and its digit rule must catch SEPARATED identifiers (practice phone `303-986-9337` = 3/3/4; DOB `04/12/1985` = 2/2/4), so the candidate rule is `\d[\d\s.\-/]{3,}\d` plus emails and `Bearer`/`token`/`key`/`secret`/`password`/`apikey`. But that digit rule OVER-redacts legitimate inputs : ISO dates (`2026-06-22`), `127.0.0.1`, version strings (`v1.2.3`), `:line-line` citations, and numeric path segments all match. A scrubber that mangles its own safe inputs gives false confidence ("the scrubber ran") while destroying legitimate text and training devs to route around it. So the test set MUST include NEGATIVE (leave-untouched) cases asserted unchanged: a clean repo path with a numeric segment, an ISO date, `127.0.0.1`, a `:line-line` citation; the regex is refined (word boundaries / min length) before wiring to any caller. The opt-in free-text branch ships DISABLED, gated behind explicit per-use confirmation IN CODE (not just prose) if ever enabled. Canned templates are the only enabled path.
- The M0b prompt-prefix redaction defaults to ID-ONLY (the zero-false-positive option, chosen), NOT the scrubber: the two pre-existing prompt-prefix log lines (`hook-router.ts:56`, `tab-namer.ts:74`) drop the raw prompt substring entirely and log id + length only. `scrubFreeText` is REJECTED for those lines (its false-positive surface is unnecessary where id-only is sufficient). The scrubber ships with its real caller only when the Phase-2/3 free-text query opt-in is built (M19).
- Logging reads action-id + repo ONLY, never the composed body (3.6). Use the `log` module (`src/main/logger.ts`), never `console.log`.
- Residual: cmd.exe argv is BOTH a quoting hazard AND a leak surface for the LATER positional-prompt optimization. Write-after-ready never makes the query a `cmd.exe` argument, so Phase 1 is unaffected. But the positional path (9.3) would put the query string in the `cmd.exe /c claude` command line, which process enumeration, Sysmon, and EDR on this PHI-server-adjacent work PC capture, so any free-text positional prompt is a PHI/secret exposure INDEPENDENT of quoting. Two reasons to prefer write-after-ready and to keep any positional path canned-only; "no free-text prompt in argv" is added to the spike's acceptance criteria (9.3).

### 3.6 Security hardening (PHI/secret leak paths)

The dashboard makes the work this app runs systematically more clinical/financial, so the pre-existing exposure paths get explicit control changes rather than a "confirm during M10" check. The logger leak (above) IS a Phase-1 milestone because the dashboard's own data ships through it. The remote-auth hardening is NOT, because it touches no dashboard code and the dashboard adds no new remote broadcast; coupling it would block ship behind an unrelated remote-access change. The distinction is the "is this dashboard work or a separate security project" test, applied consistently.

- Remote auth hardening (a SEPARATE security issue, filed not coupled, formerly C1): the remote web surface broadcasts raw PTY output (full live + scrollback of every Claude session: OD queries, patient data on screen, financial reports) to any synced client (`index.ts:82-83`, `web-remote-server.ts:115-129,253`), guarded only by a 6-char token over a 31-char alphabet (`ABCDEFGHJKLMNPQRSTUVWXYZ23456789`, verified `web-remote-server.ts:57-58`). State the keyspace honestly: 31^6 = 8.87e8, WEAK. But the token is a SECONDARY control: the real exposure is that the server, though bound to `127.0.0.1` (`web-remote-server.ts:78`), is fronted by a PUBLIC Cloudflare quick tunnel (`tunnel-manager.ts:25,103`) that mints `https://<random>.trycloudflare.com` with NO Cloudflare Access and NO IP allowlist, and the tunnel terminates traffic as localhost so the bind gives no source restriction. Anyone who learns the URL (history, phone, screenshot, Telegram) reaches the auth handshake. Widening the token helps brute force but is a band-aid for an internet-reachable endpoint that broadcasts clinical-session scrollback. This is a pre-existing hole in the already-shipped remote-access feature, exploitable today with zero dashboard code, and the dashboard adds no new remote broadcast (M5 asserts `program-board:state` is NOT forwarded). So it is FILED as its own security issue/PR against the remote-access feature (exactly as 9.1 files the `--plan` bug), with its own review, NOT smuggled into the dashboard critical path. The filed issue recommends, in priority order: (1) a NAMED Cloudflare tunnel + Access policy (the actual fix for a public-by-URL endpoint); (2) failing that, a one-line Remote Access UI warning that the URL is publicly reachable so it is not pasted casually; (3) a bounded failed-auth / connection-attempt counter in `handleWebSocketConnection`, PROMOTED to a do-now independent one-liner (it is cheap, orthogonal to the tunnel fix, and the 10s unauth timeout at `:184-190` bounds nothing against an attacker who reconnects freely to brute the 31^6 keyspace); (4) widen the token to >=16 chars from the same 31-char alphabet (`web-remote-server.ts:58`, the client echoes the issued token so no client change). ### 5.1 Inputs

### 5.2 The R2 spine, made precise (with the idle-age floor)

`idleNeedsYou` is the primary live-session needs-you signal, NOT `requiresResponse`. `hadActivity` is the main-side `firstActivityAt` field (2.2), stamped on first entry to `working`, so it survives renderer reload and rides `tabs:sync` to remote. A freshly-resumed tab that lands on `idle` with `firstActivityAt === null` is "ready", not "needs you" (mirrors `hook-router.ts:99-107`, pinned by the M1 edge test).

Idle-age FLOOR (fixes the badge-soup defect): R2 fires `idle` once per turn, so in normal multi-session use several tabs sit idle-after-activity almost constantly, and a never-low "N need you" count stops meaning anything (E4 A1/A3/A9). A tab human-waiting for LESS than the floor is "your turn, mid-flow" and stays in the SUBORDINATE strip, NOT the needs-you list or count. Only after crossing the floor does `idleNeedsYou` become true and the item enter Tier 2.

THE FLOOR IS ONE COMMITTED NUMBER: `IDLE_AGE_FLOOR_MS = 60_000` (60s), a named export in `src/shared/`, NOT "~45-60s". This is the threshold deciding both "N need you stays meaningful" (calm-by-default) and "a waiting session escalates to the hero" (accountability), and 45 vs 60 is a real ADHD difference: 45 manufactures mid-flow nags, 60+ delays the accountability chime. The plan errs toward CALM (60s) because the waiting session is visible in the strip the whole time, so a higher floor only delays promotion to the hero, never under-surfaces. Tuning is a ONE-LINE constant change guarded by the M9 floor test, NOT a settings surface; two builders cannot ship two numbers because the constant is the single source. The floor is measured against `waitingSince` (2.2), NOT `statusSince`, so an `idle -> requires_response` transition mid-wait does not reset the clock and under-age the tab that has been waiting longest. The floor predicate GUARDS null (a resumed tab has `firstActivityAt:null` and `waitingSince:null`; `now - null` yields a ~56-year age that would push every ready-but-never-worked idle tab past the floor into needs-you on the first poll, falsely elevating it and, via 4.6, falsely making it the empty-board hero). The predicate is `waitingSince !== null && (now - waitingSince) >= IDLE_AGE_FLOOR_MS`. One threshold in the mapper, gated by a test that a `firstActivityAt:null, waitingSince:null` tab is NOT `idleNeedsYou` regardless of wall-clock (M9). This is also the secondary heuristic for the AskUserQuestion mid-turn gap (no hook fires): crossing the floor escalates the tab. Runs in the app's own process, not subject to Claude Code's flaky `idle_prompt`. `requiresResponse` is an additive overlay: when the Notification hook fires it boosts an item within its tier but is never the sole gate.

### 5.3 Exact ordering (tiers, highest first)

The hero is the first item of the highest non-empty tier.

1. Tier 1: Time-sensitive due now or soon. `timeSensitive !== null && date(timeSensitive) <= now + 5 days` (the producer's needs-you window, 4.4). A hard external deadline beats everything. (Live producer today: `practice-reports`; built in M6.)
2. Tier 2: Live session waiting on you (past the idle-age floor), with the overlay boost. `idleNeedsYou === true`. Within the tier, `requiresResponse` sorts above non-`requiresResponse`, then longer-waiting (larger `now - waitingSince`, NOT `statusSince`, 2.2) first.
3. Tier 3: The 90%-killer. `dodAlmost === true` (producer-driven, 4.4). Named by `dodGap`, shown as the goal-gradient fraction (1.10). Its own tier above generic needs-you.
4. Tier 4: `needsYou` program cards (structured: `needs_you === true`, or `BLOCKER_TAGS` membership). Within the tier, hotter `ageColor` first (red > orange > yellow > green), then by `gitAgeDays`. In Phase 2 a read-only avoidance slug/name tie-break (5.4 step 4) is added so an avoidance item does not sink below an equally-aged fresher dev card; it is NOT in Phase 1 (it re-sorts the producer-mirrored list 1.11 mandates, and fires on zero live cards, avoidance tie-break dropped from Phase 1).
5. Tier 5: Phase 2 `@now` todos and avoidance-category items, pinned by category (no commits to age them). Avoidance items sort above plain `@now` todos, then by capture age oldest-first.
6. Tier 6: Everything else (active/blocked program cards, `@next`/`@later`), not hero-eligible.

### 5.4 Tie-breaks (applied in order, within any tier)

1. Hotter `ageColor` (red, orange, yellow, green).
2. `requiresResponse === true` before false (overlay).
3. Recency proxy: use `gitAgeDays`, NOT `last_touched`. DELIBERATE DIRECTION DECISION: the producer sorts `(not needs_you, git.age_days)` ascending = newest-committed first (`src/program_board/poller.py:115`). To keep the dashboard's Tier-4 order CONSISTENT with the board the user also sees, mirror the producer: `gitAgeDays` ascending (newest first). `last_touched` is rejected as a sort key because it is sometimes tz-aware, sometimes naive (a worse key). For Phase-2 capture-age items, oldest-capture-first (staleness is the failure being solved there).
4. Read-only AVOIDANCE tie-break (PHASE 2, built with `rankItems` in M6, not Phase 1): when two Tier-4 items tie on the above, a card whose SLUG/NAME matches the avoidance keyword set sorts ABOVE a non-avoidance card. This is one slug/name keyword check, never `blocked_on`, never logged, never into `composeClaudeQuery`; it is NOT the M13 classifier (which reads `blocked_on`/`needs_you` text and pins by category). M6 tests that an avoidance-slug card outranks a non-avoidance card when all prior keys tie.
5. Stable `id` lexical order (deterministic final tie-break so the hero never flickers between two equal items on re-poll; also makes the function testable).

### 5.5 Stability rule (anti-flicker)

Deterministic for a given input set; the `id` tie-break means a re-poll that changes no ranking input does not reorder the hero. The renderer keys by `id` so an unchanged hero is not remounted. Satisfies "no uninvited motion."
