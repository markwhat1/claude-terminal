Three more are calibration-of-restraint gaps the lens grades directly. None are architecture problems. They are the last measurability and day-one-ergonomics gaps, which is exactly this lens's job.

## Must-fix (blocks a 9 on this lens)

### MF-1. Phase 0 ships a hero whose one action is "copy a command and go run it yourself." Name the Phase-0 activation reality honestly, or pull the smallest real forward action into Phase 0.

The plan's headline ADHD bet is "surface the one thing AND pay the initiation cost for the user" (1.7 reframe-as-review: "Initiation cost is paid by Claude, not by Mark"). That payment is the entire point of the affordance. But it ships in M10 (Phase 1). Phase 0 (M8a, the FIRST dogfoodable artifact, the build that ships and gets looked at first per the plan's own "each phase ships and gets used first" rule) has Copy as the hero's primary action (M8a change list: "Copy is the hero's primary action in Phase 0 (canned paste-ready, 3.3) so the button is never dead before M10 wires the write actions").

For this lens that is the activation move INVERTED on day one. Copy-a-command means: read the hero, click Copy, switch to a terminal, paste, press Enter, wait for the CLI, then start. That is MORE steps than the status quo of just opening a terminal, not fewer. The plan measures "zero hero-region layout shift" and "one keystroke to first move" as Phase-0 activation numbers (1.13), but the keystroke lands on Copy, which does not start anything. So the Phase-0 build the gate's 5-day window partly runs on is a board whose one-thing hero LOWERS nothing and arguably RAISES activation energy versus the terminal the user already has open. If the gate's early days run on that, a "Home is not compelling" read is confounded by "Home's one action made me do more work."

This is the day-one analogue of the round-5 MF-3 cue-tax finding: the bet is taxed at the exact surface that is supposed to prove it.

Two honest fixes, pick one:

- **Option A (preferred for the lens): pull the Open-PowerShell action into Phase 0.** It is the lowest-plumbing of the three actions (no new IPC channel, reuses `createShellTab` + `explicitCwd`, the resolver is already specced for Phase 1 in M10, 3.2/2.5). A hero whose primary button OPENS A SHELL IN THE HERO'S OWN REPO is a real forward action: one keystroke from window-focus, the cwd is already right, no copy-paste-switch dance. That makes the Phase-0 activation numbers (1.13) describe a button that actually starts work, not one that copies a string. The reframe-as-review write action still lands in M10 as the headline; Phase 0 just stops shipping a hero that adds steps. Scope: the `resolvePreferredPowershell` helper (M7) and the `explicitCwd` `tab:create` param are the only Phase-1 pieces this pulls earlier, and `explicitCwd` already exists on `tab:createShell` today (no new param needed for the shell path), so Open-PowerShell-into-the-hero-repo is buildable in M8a/M9 territory with the M7 resolver and zero new channel. M8a's "no-action-constructible -> Copy-only hero" fallback (6.3) stays as the genuine fallback when no repo resolves.

- **Option B (if A is too much for Phase 0): state plainly that Phase 0 is a READ-ONLY GLANCE board with no activation claim, and move the 1.13 keyboard-floor / one-keystroke activation framing to Phase 1 (M10) where the keystroke actually starts work.** Then Phase 0's only honest activation number is the layout-shift one (a glance guarantee, not an initiation guarantee), and the plan stops crediting "one keystroke to first move" to a Phase-0 hero whose first move is Copy. The gate's early days then explicitly do not test the activation bet, only the glance bet, and the gate language says so.

Either is fine. What is not fine on this lens is shipping a Phase-0 hero billed (1.13) as a one-keystroke activation win whose one keystroke copies a command the user must then go run by hand. Recommend Option A: the cheapest real forward action on the calmest surface is the single highest-leverage activation change left in the plan, exactly as R-10 Option A was last round for the cue.

### MF-2. The Phase-0 time-to-glance has a layout-shift number but no STATE-SETTLE number. Bound the cold-open-to-readable-hero wait, or the calmest screen still makes the time-blind brain wait an unbounded beat before the one thing is legible.

1.13 names "zero hero-region layout shift between skeleton and content" as the Phase-0 time-to-glance guarantee, and M8a/4.5 test that the skeleton and content share the hero footprint. Good, that kills the reflow jump. But layout-shift-zero is not the same as time-to-legible-hero. The first-open timeline (4.3) holds the skeleton until the first read resolves, and the read is the ~20s poll's first tick OR the first successful file read. On a cold open the user can sit on a SKELETON (a gray box where the one thing will be) for an unbounded interval, because the plan's only Phase-1 freshness mechanism is the ~20s poll (R-2, watcher deferred) and the cold-file HTTP fallback is also deferred (4.3 item 3). For a time-blind ADHD brain, a skeleton where the hero should be is functionally "nothing is here yet," the now-vs-not-now disengagement trigger the plan cites elsewhere (E3, 1.5b). The plan specified the success-PENDING surface for the multi-second CLI-boot window (1.5b) precisely because silence reads as nothing-happening, then left the cold-open skeleton-to-first-read window with no equivalent bound or number.

The fix is a number plus a behavior, both cheap:

- **State the cold-open budget: the hero is legible (skeleton replaced by content OR by a committed empty/error state) within ONE poll interval worst case, and immediately on the synchronous first read when `state.json` already exists (the always-on nssm common case).** The second half is the important one for the lens: on the overwhelmingly common path (the service is up, the file exists), M4's FIRST read is synchronous-ish on construction, not gated on the ~20s tick, so the hero should paint on first reader construction, not after a poll cycle. Make that explicit: the reader does an IMMEDIATE first read on construction (not "wait for the first poll tick"), so a warm open paints the hero in one read, and only a genuinely-down service shows the skeleton-then-"not running" path. State the number ("warm open: hero legible on first read, no poll-cycle wait; cold/down: bounded to one poll interval then the committed empty state") and put it in M4 / M8a's DoD next to the layout-shift number.

- **Encode it as a test the way layout-shift is encoded.** M4 already uses a real temp file; assert the reader emits its first parsed state on construction (an immediate read), not only on the first interval tick, so a builder cannot ship a reader that makes the calmest screen wait 20s for its one thing on every warm launch. M8a's fake-timer first-open test (4.3 item 3) already drives the skeleton-persists case; add the warm-file case: with `state.json` present at mount, the hero content paints WITHOUT waiting a poll tick.

Without this, the Phase-0 activation story is "no reflow jump once it arrives" with no bound on WHEN it arrives, and on a time-blind brain an unbounded wait for the one thing is the activation tax the whole surface exists to remove. This is the second half of the time-to-glance metric the plan opened (1.13) and did not finish.

## Top improvements (raise toward 9)

### TI-1. The done-lane carrot now rarely fires AND its louder tier is Phase 2, so Phase 1 ships a "reward" surface that is mostly invisible. Decide the open-question-14 "decided and worked" exception NOW so Phase 1 has at least one carrot that actually pays.

The plan is admirably honest here (1.5, 1.12, R-8): on live data 4 of 6 needs-you cards close via a removed override tag (a decision made, a block lifting), which the progress guard EXCLUDES, so the one real reward "today mostly does not pay" (1.12). And the louder two-tier beat rides M13 in Phase 2. Stack those and Phase 1's restrained-dopamine axis is: one reward, one tier, and that one tier fires almost never. For a low-baseline-dopamine brain that is functionally no carrot for the whole Phase-1 window, which is the window the gate runs on.

The plan offers the fix and then defers it: open question 14's "decided and worked" exception (count a `needs-your-decision` tag clearing as a finish when the card is NOT a simultaneously-lapsing `time_sensitive` AND `last_commit.iso` is within ~1 day). That exception is exactly calibrated, it separates "decided and worked" from "silently deleted the tag," and it is the difference between the carrot paying on Mark's actual dominant win (resolving a parked decision, the literal avoidance-loop close) versus paying almost never. Deferring it to a sign-off means the most likely Phase-1 outcome is a reward surface that the gate-window user never sees fire, which on this lens reads as "the finish does not pay out," the exact E5-P2 failure the carrot exists to prevent.

Recommend: make the "decided and worked" exception the Phase-1 DEFAULT in M4b (not just an offered sign-off), guarded exactly as 1.5 states (`last_commit.iso` within ~1 day AND not a simultaneously-lapsing deadline), with the M4b test asserting it fires for a decided-and-committed card and does NOT fire for a bare tag deletion. That keeps the lapse-never-pays honesty (the over-fire guard stays) while giving Phase 1 a carrot that pays on the real win. If Mark prefers to keep it as a sign-off, then 1.12 and the gate language must say plainly that Phase 1 may ship with a NON-FIRING reward surface, so the gate does not read a "no felt reward" result as "the board is not compelling."

### TI-2. The avoidance tie-break is proposed but not committed, so Phase 1 may ship with ZERO avoidance presence, which is the hollow-axis failure round 5 flagged and round 6 only half-closed.

The lens grades "shame-free avoidance nudges." The plan handles shame-free well (no OVERDUE, capped heat, neutral verb-first copy, the voice test banning still/again/keep). But the avoidance PRESENCE in Phase 1 is still gated behind an OPEN QUESTION (12), not committed. 1.4/5.4 specify the read-only slug/name tie-break beautifully and constrain it correctly (never `blocked_on`, never logged, never into `composeClaudeQuery`), but every reference frames it as "PROPOSED" / "the plan now proposes pulling" / "confirm pulling that tie-break." If Mark does not confirm, Phase 1 ships a board that is shame-free precisely because it surfaces no avoidance items at all (the one present, `marketing-roi`, is paused and filtered). That is the hollow version of the axis R-13 names as the most-likely-stranded value.

This is cheap to commit and it is the cheapest way to make the foundation hint at the eventual value with real behavior. Recommend: commit the slug/name tie-break as the Phase-1 DEFAULT (5.4 step 4, M8a-tested as already specced), scoped exactly as written, and downgrade open question 12 to "here is the keyword set, confirm or edit the list" rather than "confirm whether to do this at all." The keyword set is the only real decision (it is Mark's six avoidance areas: financial, documentation, delegation, completing-the-loop, health, marketing); the mechanism is already designed, tested, and PHI-safe. Shipping the mechanism OFF by default makes the gate measure a board with no avoidance presence, which cannot tell Mark whether avoidance surfacing would have earned the daily open. Commit it; let the keyword list be the tunable.

### TI-3. The idle-age floor is a single hardcoded ~45-60s constant with no calibration path, and it is the gate between "calm periphery" and "your turn, look now." Commit one number AND state how it gets tuned without a code change.

The idle-age floor (5.2) is load-bearing for two of the lens's axes at once: it keeps "N need you" meaningful (calm-by-default, no badge soup) AND it is the threshold at which a waiting session escalates from the subordinate strip to the needs-you hero (the present-partner accountability axis, 6.4). The plan says "~45-60s floor" in 5.2 and "the 45-60s idle floor ... need a real-usage tuning pass" in open question 2. That is the same "around 4-5" vagueness the plan correctly refused for the disclosure cap (4.6 committed N=4) and the contrast tokens (6.2), left un-refused for the one threshold that decides whether the calmest screen is calm or a nag.

Two builders will ship 45 vs 60, and on this lens the difference is real: 45s surfaces mid-flow "I just stepped away" pauses as "needs you" (over-nag, the badge-soup failure), 60s+ lets a genuinely-waiting session sit in the periphery a beat too long (under-surfaces the accountability loop, the chime Mark used to listen for, 6.4). Recommend: commit ONE number in the mapper (recommend 60s: erring toward calm, since the strip still shows the waiting session the whole time, so a 60s floor under-surfaces nothing, it only delays PROMOTION; an over-eager 45s floor actively manufactures nags). Put the constant in `src/shared/` as a named export, and state that tuning is a one-line constant change with the M9 floor test as the regression guard, not a settings surface (it does not earn one). One number, one named constant, one stated tuning lever. This closes the last "around N" in the activation-critical path.

### TI-4. The success-pending line is calm but says nothing the time-blind brain can act on. Give it a coarse, honest progress cue without a metronome.

1.5b is the right instinct: the multi-second CLI-boot window after the hero click gets a "Starting your session, the first step will be typed in for you" line on the spawning tab, sub-dominant, no working-spinner, reduced-motion-static. Good. But a STATIC line for a window that can run 3-8 seconds (the plan's own estimate) is, to a time-blind brain, indistinguishable from a hung line at second 7. The plan banned the `Loader2` spinner (correctly, it reads as "Claude is working" on an empty pane), but it left the window with NO temporal cue at all, which trades one wrong signal (working) for another (nothing-happening), the exact E3 disengagement trigger 1.5b cites as its reason to exist.

The fix is a calm, honest, NON-metronome cue. Recommend: a single coarse heartbeat that is motion-safe and NOT a continuous animation. Either (a) the line itself coarsens its own copy at a slow, irregular-feeling step ("Starting your session..." -> at ~4s "...still starting, this can take a few seconds"), a ONE-step copy change at a single threshold, no loop, reduced-motion shows the second form statically if the window exceeds the threshold; or (b) a single slow non-looping motion-safe fade-in of a second reassurance line at ~4s. Both are one threshold, one copy/class change, no metronome, and both tell the time-blind brain "still going, not hung." The 30s failed-start surface (3.1 step 7) already owns the hard timeout; this fills the legibility gap in the 3-8s SUCCESS window between "armed" and "ready" so the user does not disengage at second 6 and undo the reframe in its first beats (1.5b's own stated failure). M10 already tests the pending affordance shows-and-clears; extend it to assert the second-form copy/line appears after the threshold under fake timers and is reduced-motion-safe.

### TI-5. The "N need you" glance number is the board's primary calm-signal, but its honesty rules are spread across four sections. Pin a single "the glance number never lies" invariant with one consolidated test, the way the restraint inventory consolidated the dopamine claim.

The plan worked hard to make the glance count honest: it is computed off the UNIFIED set so it never reads "0 need you" above a waiting hero (4.6); the idle-age floor keeps it meaningful (5.2); paused cards are excluded (4.4); the closed count reads "last 24h" not "today" (1.5). Each is correct and each is tested in its own milestone. But "the single number a time-blind brain trusts at a glance" is the load-bearing calm-by-default signal, and right now its correctness is an emergent property of four separate guards in four sections, exactly the situation the restraint inventory (1.12) was created to fix for the dopamine claim. A contributor who changes the unified-set computation, or the floor, or the paused filter, can break the glance number's honesty without any single test framing it as "the glance number now lies."

Recommend a small consolidation, mirroring 1.12's move: a one-paragraph "glance-number honesty invariant" (the count equals exactly the things that genuinely need Mark right now: unified board-needs-you + past-floor waiting tabs, minus paused, and the closed count is an honest rolling-24h, never a since-midnight reset) plus ONE consolidated M8b assertion that drives the adversarial case (programs empty + one past-floor waiting tab + one paused needs-you card + a nonzero closedRecent) and asserts the header reads "1 need you" and "N closed, last 24h" with no "0" and no "today." The pieces are all already tested; this adds the single test that proves they compose into an honest glance, so the one number the calm board lives or dies on cannot silently start lying. It is the calm-axis analogue of the restraint inventory: make the honesty auditable in one place, not asserted across four.
