## Defects

### D1 (HIGH) The "Phase-1 avoidance tie-break" is simultaneously decided, open, milestone-homeless, and contradicted by the mirror-the-producer rule

This is the round-6 version of the recurring seam defect. The read-only avoidance slug/name tie-break appears in five places that do not agree:

1. **Decided, shipping in Phase 1.** Section 1.4 (line 101): "A READ-ONLY avoidance keyword TIE-BREAK ships in PHASE 1." R-8 (line 1028) lists it as a Phase-1 mitigation in the present tense ("the read-only avoidance tie-break pulled into Phase 1").
2. **Still an open question.** Open question 12 (line 1051): "Confirm pulling that tie-break into Phase 1, OR accept Phase 1 reading as a sharper status board first with avoidance surfacing fully deferred to M13." So it is NOT decided; it is a thing Mark must still approve. A plan cannot state a behavior "ships in PHASE 1" (line 101) and "PROPOSES pulling [it] into Phase 1... Confirm... or accept [deferral]" (line 1051) as the same fact.
3. **No clear milestone home.** Line 640 says "M8a/M6 test that an avoidance-slug card outranks a non-avoidance card." Line 631 says the Phase-1 single-field override "carries this one tie-break." But M8a's actual change/test list (lines 896-897) contains ZERO avoidance assertion: the override M8a builds is "time-sensitive within 5 days, else dodAlmost, else producer needs-you head" (line 896), with no Tier-4 tie-break machinery at all. So line 640's "M8a... test" claim has no backing in M8a's own milestone body.
4. **Classified as Phase-2/M6, then Phase-2/M13.** Section 5.6 lists "an avoidance-slug card outranks a non-avoidance card when all prior Tier-4 keys tie" under "**Phase-2 assertions (built in M6)**" (line 651), and the very next line (652) lists "the avoidance-pin classifier tie-break (M13)" under "Deferred (built with their producers)." So 5.6 puts the same behavior in M6 AND defers it to M13, while 1.4/R-8 put it in Phase-1 M8a. Three milestone homes (M8a, M6, M13) for one tie-break.
5. **Contradicts the explicit no-re-sort rule.** Section 1.11 (line 192): "slots 1..n deliberately match the producer so a Phase-2 builder does not re-sort the list and diverge from the board." An avoidance tie-break that promotes an avoidance card ABOVE an equally-aged non-avoidance card in Tier 4 IS a re-sort that diverges from the producer board order the user also sees. The plan forbids exactly this re-sort one section away from mandating it.

Where it fails: this is not a documentation nit. A builder executing M8a builds the override (time-sensitive/dodAlmost/producer-head) per lines 896-897 and ships green WITHOUT the avoidance tie-break, because M8a's test list never asserts it. Meanwhile 1.4/R-8 have told Mark the Phase-1 board "hints at the eventual value" via that tie-break, so the one avoidance-facing Phase-1 signal R-13's mitigation rests on (line 1034: "Phase 1 carries at least one avoidance-facing signal (the read-only avoidance tie-break)") silently does not ship. R-13's whole roadmap-risk mitigation is load-bearing on a feature that has no milestone that builds it.

Minimal fix: pick ONE answer and make all five sites agree. Cleanest given the YAGNI lens and the no-re-sort rule (1.11): DROP the avoidance tie-break from Phase 1 entirely. It is the M13 classifier's job (Phase 2), it re-sorts the list 1.11 says must mirror the producer, and on live data the only avoidance item (`marketing-roi`) is paused-and-filtered so the tie-break fires on ZERO live cards in Phase 1 anyway (no two non-paused avoidance-vs-non-avoidance Tier-4 cards exist to tie). Make open question 12 a clean "accept Phase 1 is a sharper status board, avoidance surfacing is Phase-2 M13," delete the line-101 "ships in PHASE 1" claim, delete the line-640 "M8a... test" claim, drop it from 5.4 step 4 / Tier 4 (lines 631, 640) for Phase 1, and rewrite R-13's mitigation (line 1034) to rest on the goal-gradient pull alone (which actually ships in M8a). If Mark instead WANTS it in Phase 1, then it must become a real M8a (or a new tiny M8a-adjacent) milestone with its own test line, and 1.11's no-re-sort rule must carve out the exception explicitly. Either way: one home, one decision, five consistent sites.

### D3 (MEDIUM) M10 is a phase wearing a milestone's badge; "one change, one test, one rollback" is violated and the plan half-admits it

The plan's milestone discipline is stated as a hard rule (line 64, line 762): "Milestones are boringly small: one change, one expected vitest test, one rollback point." M3a was correctly split into three for exactly this reason (line 832: "round-5 M3a was a phase disguised as a milestone... violating one-change/one-test/one-rollback"). M10 is the same violation, un-split. Its change list (lines 921-928) is: `composeClaudeQuery` + `composeCopy` + `pickPrimaryAction`; the `explicitCwd` `tab:create` param + `resolveProgramProject`; the `claude:injectQuery` handle channel; the `claude:injectStatus` broadcast channel; the MAIN pending-injection Map + once-flag + 30s timeout; the `HookRouterDeps` extension giving MAIN `ptyManager.write`; the convergence-point idle gate; toast suppression on injected first-idle; the explicit `bypassPermissions` override; the 1.5b pending affordance on the spawning Terminal surface; the failed-start retry; the pwsh resolver; Copy. Its test list (lines 929-938) is ELEVEN distinct assertions across MAIN IPC, hook routing, the PTY write, two new channels, and renderer pending UI. That is more surface than M3a had when the plan split M3a for being a phase.

The plan half-admits this: line 919 gives M10 an "internal sub-sequence for intermediate rollback points... (1) composeClaudeQuery + pickPrimaryAction, (2) the explicitCwd param, (3) the channel pair + MAIN gate + timeout, (4) the pwsh resolver + Copy" with "/cad:milestone after each." So the plan KNOWS M10 is four milestones; it just labels them "sub-points of M10" instead of M10a/b/c/d. That is the M3a defect the plan fixed, re-committed under a different name. A "sub-sequence" with four commit points is four milestones; calling it one milestone means M10's single DoD (lines 939) gates four independent changes, so a reviewer cannot sign off piece (1) without piece (4) being done.

Where it fails: M10 is the most-coupled, highest-risk milestone in the plan (net-new cross-module MAIN feature: HookRouterDeps contract change, two new channels, a pending-state machine, the headline 1.7 affordance). It is exactly the milestone that most needs the boringly-small discipline, and it is the one that gets a four-in-one exemption. The arm-before-resolve property, the convergence-point idle gate, and the toast suppression are each subtle enough to deserve their own rollback point and their own green/red signal.

Minimal fix: split M10 into M10a-d on the sub-sequence the plan already names (line 919), each with its own one-change/one-test/one-rollback, exactly as M3a was split. M10a (`composeClaudeQuery`/`composeCopy`/`pickPrimaryAction`, pure, no IPC), M10b (the `explicitCwd` `tab:create` param + `resolveProgramProject`), M10c (the channel pair + MAIN gate + timeout + HookRouterDeps + toast suppression + pending affordance, the genuinely coupled core), M10d (pwsh resolver + Copy + the permission override). M10c is still large but is the irreducible coupled unit; M10a/b/d peel off cleanly. This also makes the Phase-1 ship criteria (lines 60, 965) reference real testable rollback points instead of one mega-milestone.

## What is NOT a defect (to preempt counterarguments)

- The round-5 HIGH (M4b on the Phase-0 critical path) is genuinely fixed: line 777 explicitly states "M4b is NOT on this path" and moves it to Phase 1 beside M8b. I am not re-raising it.
- The architecture is sound and not over-engineered. Home renderer-only (2.1), the single new local IPC channel for the reader (M5), the poll-primary/watcher-deferred consumption (4.3, an honored R4 departure flagged at 2.11), and the `rankItems` Phase-2 deferral (Section 5 "Phase split YAGNI") are all correct YAGNI calls. The single-field override (1.11) replacing the tiered ranker in Phase 1 remains the model restraint move.
- The M3a -> M3a-i/ii/iii split is a real improvement and the right precedent. My D3 is that the same split was NOT applied to M10, not that M3a is wrong.
- The carrot honesty is now genuinely good. 1.5 and 1.12 admit plainly that the done-lane carrot rarely fires on live data (the dominant close path is excluded) and explicitly stop banking it as Phase-1 weight. That is the round-5 D5 fix landed honestly, not relabeled.
- The program-board seam is still clean; the app owns no producer code and `closed.json`/`todos.json` live under `userData`, out of the workspace git tree. Not two-products-fused.
- The security/PHI scope boundary (3.6: logger leak IS Phase-1 M0b because the dashboard's own data ships through it; remote-auth hardening is CUT and filed because it touches no dashboard code) is the correct dashboard-vs-separate-project test, applied consistently. C1 being cut to a filed issue is the right scope call, not scope-dodging.

## Summary

Round 6 closed the decisive round-5 scope defects (M4b off the critical path, the ship-criteria reconciliation, the carrot honesty, the stale pointer) and the M3a split is a real win. The architecture is not over-engineered and not two-products-fused. But round 6 repeated the pattern: it opened a new contradictory seam (the Phase-1 avoidance tie-break, decided/open/milestone-homeless/re-sort-forbidden across five sites, D1, HIGH), made the over-documentation defect worse not better (1085 lines, +83, "fits in one head" still claimed and still false, the round-5 D4 extraction not done, D2, MEDIUM), left the highest-risk milestone M10 as a four-in-one phase-wearing-a-milestone-badge that the plan's own sub-sequence admits is four milestones (D3, MEDIUM), and padded the open-questions list with already-decided items (D4, LOW). Fix D1 (drop or properly milestone the avoidance tie-break) and D3 (split M10 like M3a) and the Phase-0/1 build path is honest and shippable. D2 is the meta-defect: until the plan actually fits in one head, the next round will hide the next contradiction the same way. The plan is close; it is over-DOCUMENTED, and the over-documentation is once again the thing hiding the small contradictions.
