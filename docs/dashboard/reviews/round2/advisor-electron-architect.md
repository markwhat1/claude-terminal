## Findings (ordered by build-day impact)

### Finding 1 (citation precision): the render-seam line range fights R5's own correction

The plan cites the render seam as `App.tsx:576-583` throughout (Sections 2.2, 2.8, M3a, appendix). R5 §2 explicitly CORRECTED the prior recon: "App.tsx Terminal render entry is at `:578-581` — prior A doc said `:577-583`." The actual code: container `:576`, `tabs.map(` at `:577`, `<Terminal .../>` at `:578-581`, map close `:583`. The plan's `:576-583` is the container-inclusive span, which is defensible as "the block the sibling sits next to," but it silently re-adopts the exact number R5 flagged as the wrong-revision artifact, and R1 §(a) independently says `:577-583`. For a plan whose headline is "every file:line verified against this exact checkout," shipping the number a sibling R-doc just corrected is the kind of thing the lens grades down. Fix: cite the sibling-insertion point as "after the `tabs.map` block, inside `data-terminal-area` (`App.tsx:576` container, map `:577-583`)" and drop the bare `:576-583` range so it reads as deliberate, not a stale copy. Reconcile the R1 (`:577-583`) vs R5 (`:578-581`) framing in one sentence so the build lane is not left guessing which R-doc wins on this one number.

### Finding 2 (buildability gap, will silently no-op): `Ctrl+Shift+K` must register as uppercase `'K'`

Both R6 item 3 and Section 8.6 declare `Ctrl+Shift+K` "clean" and land it in the `ctrl+shift` matcher arm. True for collision. But neither says what key STRING to register, and the matcher at `keybindings.ts:77` compares `e.key === kb.key` with case-sensitive single chars. Every existing entry is lowercase (`'n'`, `'t'`, `'w'`, `'p'`). When Shift is held, `KeyboardEvent.key` for that physical key is the uppercase `'K'`, not `'k'`. A registry entry `{ mod: 'ctrl+shift', key: 'k', ... }` will NEVER match, because `e.key` is `'K'`. The existing `Ctrl+Shift+Tab` entry (`key: 'Tab'`) dodges this because `Tab` is case-invariant. So the capture bar would ship dead with green-looking code. Fix: M12 must register `key: 'K'` (uppercase), and the M12 test must dispatch a `KeyboardEvent` with `key:'K', ctrlKey:true, shiftKey:true` and assert the action fires (not `key:'k'`). State this explicitly in 8.6 and the M12 test surface; it is the single most likely "passes review, fails in the app" defect in the plan.

So "assert registered" concretely means: add `'program-board:getState'` to the `expectedHandlers` array, and for the broadcast assert via a `sendToRenderer` spy (it is not an `ipcMain.handle`, so it will not appear in `handlers`). For the capture channel, add its `noun:action` name to `expectedHandlers` and, because it is remote-enabled, add a SECOND assertion that `WebRemoteServer.handleMessage` has a `case` for it (a different test file / mock than this one). The plan's "the absent-from-forward-list assertion is the tested remote decision" (M5) needs a concrete hook: `index.ts sendToRenderer` (`:75-99`) has no exported forward-list constant today (verified: it is inline `case` logic, appendix confirms the forwarded set `pty:data, tab:updated, tab:removed, pty:resized, tab:switched, tab:worktreeProgress`). So "assert NOT in the forward list" cannot be a simple array-membership check; either refactor the forward set into an exported constant (one small, justified change, worth a milestone note) or assert behaviorally that a `program-board:state` emit does NOT call `webRemoteServer.broadcast` via a spy. Pick one and name it, or M5's central remote-decision test is unbuildable as written.

### Finding 4 (security control, signature precision): the M0b logger gate must not regress the file sink

M0b says "gate `logger.ts:42-51` emit's renderer mirror to warn/error only (or behind a dev flag); keep file logging." The verified `emit(level, msg)` does `writeToFile(level, msg)` FIRST (`:43`), THEN the `executeJavaScript` mirror (`:44-50`), with an `else pending.push(...)` branch when the window is absent. Two precision points the plan should bake into the M0b change so it is one clean rollback point:

1. The gate must wrap ONLY the `executeJavaScript` block, leaving `writeToFile` unconditional, or info/debug stops hitting disk too (the plan wants file logging kept). The `else { pending.push(...) }` branch is the pre-window buffer that gets flushed on window-ready; if you gate the mirror, decide whether gated-out levels still buffer (they should not, or warn/error pending replays will be diluted). State: gated-out levels skip BOTH the live mirror and the pending buffer.
2. The M0b test should assert the file sink still receives debug/info (spy on `writeToFile`) AND the mirror does not (spy on `webContents.executeJavaScript`), so the test pins both halves of "gate the mirror, keep the file." The plan currently only asserts the mirror half.

Also: the plan repeatedly says the dashboard's own log lines carry "action-id + repo only." Good. Add the corollary M0b assertion that `composeClaudeQuery` and the `DashboardItem` mapper never call `log.*` with `title`/`detail`/`blocked_on`/`name` content. The plan mentions this in 3.6 prose; make it a named M0b test bullet, not prose, since it is the actual PHI control.

### Finding 7 (the spine's testability boundary): `firstActivityAt` survives reload but the renderer never re-derives it, good; name the one failure mode

`firstActivityAt` (the `hadActivity` spine) is stamped in MAIN on first `working` (2.2 snippet) so it survives renderer reload and rides `tabs:sync` to remote. Correct and important. The one unstated failure: a tab that was `working` BEFORE the dashboard ships (an already-running session at upgrade time) has `firstActivityAt: null` forever, because the stamp only fires on a TRANSITION into `working`, and a tab already in `working` will next transition to `idle` (no stamp). Such a tab can never enter Tier 2 until it cycles back through `working`. This is benign (it self-heals on the next turn) but it is a real edge the test surface should name, so a reviewer does not later "fix" it by stamping on idle (which would break the spine). Add to M1: "a tab whose first observed status is `idle` keeps `firstActivityAt: null` (ready, not needs-you) until it next enters `working`," with a test.

### Finding 8 (R4 consumption, one real-data caveat): the "almost done" gap label can be empty

Section 4.4 lifts `dodGap` from the producer's `"almost done: {dod['gaps'][0]}"` reason. Verified the producer does `dod['gaps'][0]` with no guard that `gaps` is non-empty WHEN `total - met == 1`. The producer's own `evaluate_dod` builds `gaps` from unmet items, so `total - met == 1` implies exactly one gap and `gaps[0]` is safe ON THE PRODUCER. But the dashboard parses a STRING ("almost done: <label>") and the label after the prefix could in principle be empty or contain a colon. The plan says "lift the gap label from the string" but does not specify the parse. Make M4's parity test assert the parse handles "almost done: " with a trailing label AND degrades to `dodGap: null` (still `dodAlmost: true`) if the label is empty, so the hero renders "one step left" even when the label is missing. The structured fallback the plan already names (`dod.gaps[0]` directly) is the better primary; prefer reading `dod.gaps[0]` from the structured object and use the reason string only to confirm `dodAlmost`, rather than string-parsing the label out of prose. That inverts the plan's stated preference and is more robust.

### Finding 10 (open question that is really a decision): the `tabs:sync` reconnect landing is described twice with slightly different words

2.1 says remote reconnect "intentionally lands on the first real tab, or a renderer-side Home sentinel the remote app computes from the forwarded tabs." 3.5 row 1 says "reconnect lands on first real tab (no Home sentinel over the wire)." These are not contradictory but the "or" in 2.1 leaves the remote-Home behavior as two options, while 3.5 commits to one. The lens cares that Home stays renderer-only AND remote-safe; the cleanest statement is: the `'home'` TabType never crosses IPC, the remote client may synthesize its own Home from forwarded tabs, and main's null `activeTabId` after last-close is expected (documented, not a bug). The plan says all of this; just deduplicate to one wording.
