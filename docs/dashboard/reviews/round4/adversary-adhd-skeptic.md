## D1 (HIGH) — The goal-gradient, billed as "the cheap dopamine win," renders ZERO progress on every live needs-you card; on real data it is a wall of "Start the first step," not endowed progress

Where: Section 1.10 (titled "Endowed progress / goal-gradient (Phase 1, the cheap dopamine win)"), Section 1.32 honesty paragraph ("It ships seven small ADHD behaviors (goal-gradient ...)"), Section 6.3 (CardContent goal-gradient line), R-8 ("the goal-gradient pull with the goal-at-zero gap frame"). Live-verified this pass: all 6 needs-you cards are `dod.met:0` (`cad-staff-portal` 0/3, `incomplete-notes` 0/1, `marketing-roi` 0/2, `od-query-consolidation` 0/3, `practice-reports` 0/2, `cad-document-pipeline` 0/3). Zero of them have `dodMet > 0`.

Why it fails the lens: 1.10's whole pitch is endowed progress, E5 P5 "never show a goal at zero," the dopamine of "4 of 5 done, last step: X." But that honest-fraction frame only renders when `dodMet > 0` (1.10 first bullet). The plan correctly forbids "0 of N" and routes `dodMet === 0` to the gap-led "Start the first step: <gap>" frame (1.10 second bullet, good, that part is right). The problem is what the live data does to the FEATURE: on every current card, `dodMet === 0`, so the goal-gradient NEVER fires its endowed-progress branch. What actually renders on the hero and every DoD row today is "Start the first step: <gap>" / "Start with: <gap>, then N more". That is not a dopamine win. That is a to-do list that says "start" six times. The plan's restraint paragraph (1.32) explicitly counts the goal-gradient as one of the "real motivational weight" behaviors carrying the calm layer; on real data it carries none, because endowed progress requires partial completion the producer is not emitting for any needs-you card.

This is the same class of defect as my round-3 D4 (the carrot that the data mostly cannot trigger), one level up: a motivational mechanism the plan banks on, structurally starved by how Mark's programs actually look (decision/blocker cards sit at met:0 until the decision is made, then they leave needs-you entirely, they do not tick 1/3 -> 2/3 -> 3/3 while still needing him). The endowed-progress dopamine the plan promises basically cannot happen on this board's data shape.

Minimal fix: no code change, an honesty correction so Mark signs off on what the goal-gradient actually does on his data. (1) In 1.10 and 1.32, state plainly: "On current data every needs-you card is `dodMet:0`, so the goal-gradient renders the gap-led 'start' frame, not the endowed-progress fraction; the honest-fraction dopamine only appears if a card ever sits partially-done while still needing you, which this board's decision/blocker cards rarely do." (2) Stop counting the goal-gradient as motivational weight in the 1.32 restraint list until the data can trigger it; reclassify it honestly as "anti-demotivation (never a bare zero)," which IS what it delivers, rather than "dopamine win," which it does not. The gap-led frame is genuinely good (it kills the "0 of 3" mountain); just bill it as the calm floor it is, not the reward it is not. This also feeds R-8, which currently lists the goal-gradient as a Phase-1 payoff it is not.

## D2 (HIGH) — The dodAlmost hero tier fires for exactly one card today, and that card's "almost done" is a single 0/1 item whose only deliverable lives in the forbidden PHI slot, so the 90%-killer hero is "almost done" with no shown progress and a generic button

Where: Section 1.11 (override branch 2: "else if any candidate needs-you card is `dodAlmost`, the one with fewest remaining steps is the hero"), Section 5.3 Tier 3, Section 4.4 (the producer predicate `dod.total - dod.met === 1`, no `>= 2` guard), Section 1.10 (the 90%-killer "turns a demand into a pull"). Live-verified: the ONLY dodAlmost card is `incomplete-notes` (`dod{met:0,total:1}`, its single gap is "portal Incomplete Notes surface live end to end").

Why it fails the lens: the plan markets Tier 3 / the dodAlmost branch as the 90%-killer, the "one step from closing this" pull (1.10). But the producer's own predicate (correctly consumed, 4.4) flags `total:1, met:0` as "almost done," and on live data the ONLY almost-done card is exactly that single-item case. So the 90%-killer hero today is a card that is 0% done (met:0), displayed as "almost done," with the gap-led "Start the first step: portal Incomplete Notes surface live end to end" frame. "Almost done" + "start the first step" + 0 of 1 is a contradiction the time-blind brain will feel even if it cannot name it: the tier promises a near-finish payoff and shows a not-started task. And per the plan's own honesty in 1.7/3.4, the real deliverable for that card sits in `dod.gaps[0]`, the forbidden free-text slot, so the canned button is "Draft the first version" against the slug "Incomplete Notes", a near-meaningless query the plan already admits is "safe but generic." Net: the single card that drives the plan's headline 90%-killer tier on real data is the weakest instance of it.

This is not the dodAlmost LOGIC being wrong (4.4 is right to match the producer, and dropping the single-item case would silently hide a real needs-you item, the exact bug 4.4 fixed). It is the FRAMING being wrong for the calm/shame-free lens: calling a met:0 card "almost done / the 90%-killer" sets a near-finish expectation the data cannot honor, and an ADHD user who clicks the "almost done" hero and finds a not-started, generically-actioned task learns the hero lies. That is precisely how a board becomes "another shiny thing I stop trusting."

Minimal fix: split the user-facing framing from the producer predicate, no logic change. (1) The hero/Tier-3 copy must NOT say "almost done" or imply near-finish when `dodMet === 0`; for the single-item-DoD met:0 case it reads as a clean "one step: <gap>" (already the gap-led frame, just do not wrap it in 90%-killer language). Reserve "almost done"/near-finish copy for `dodMet > 0` cards. (2) Add one line to 1.10/1.11 and R-8: "On current data the only dodAlmost card is a 0/1 single-item DoD, so the Phase-1 90%-killer hero shows a one-step start, not a near-finish; the endowed-progress version needs a partially-done card the board does not currently produce." (3) M8a's existing single-item-DoD assertion (it already checks the gap-led frame and the absence of "0 of 1 done") should ALSO assert the rendered hero copy for the met:0 case contains no "almost done" / near-finish string, so the framing cannot drift back. One test line.

## D3 (MEDIUM) — `startupView:'home'`, the setting the plan actively encourages, lands the ADHD user on TWO gates every launch (StartupDialog modal, then Home); R-10 names the friction but the plan never removes it for the case it is steering toward

Where: Section 7 / M14a-c (the `startupView` opt-in, "the dashboard wants to encourage"), Section 2.9 / 3.1 step 2 / 6.5 (the no-active-project Home case), R-10 ("Home + StartupDialog double-gate friction. Mitigation: Home supplements StartupDialog; `startupView` defaults to `'lastSession'`"). Verified in code this pass: `App.tsx:294` runs `startSession(cliDir, ...)` which requires a directory chosen via StartupDialog (`App.tsx:530-538`) before ANY `setActiveTabId` site (`:310/:334/:517`) the `startupView` branch hooks into. There is no path to Home that does not first clear the directory/permission modal.

Why it fails the lens: the plan's stated direction is to get Mark to `startupView:'home'` (1.6: "Phase 1's Home-on-open is the unstyled precursor of the Phase-3 morning intake"; the whole phased bet in the "What round 4 changes" gate is "a calm board earns daily opens"). But for the user who opts into Home, every single launch is: a directory/permission-mode modal (StartupDialog, a decision), THEN Home. The morning cue the plan wants to own (E7 P5, cue-bound implementation intention at app-open) is preceded by a forced unrelated decision. For an ADHD user, a decision-gate before the calm surface is exactly the activation-energy tax that makes "I'll just not open it" win. R-10 mitigates the wrong thing: "defaults to lastSession" protects the user who did NOT opt in, but the friction the plan is steering Mark toward (the `'home'` user) gets a double-gate on every launch, and R-10's "Home supplements StartupDialog" restates the problem as the mitigation. The recon's own F1 lesson is "build a working surface in the tab strip, not a splash that hijacks startup"; a modal-before-Home is a splash-before-the-surface.

Minimal fix: state the constraint honestly and pick a real mitigation, not a restatement. Either (1) when `startupView:'home'` and a last-session directory exists, auto-resume that directory (skip the StartupDialog, go straight to Home) so Home is genuinely the first surface for the opt-in user, with the dialog reachable on demand; OR (2) if the StartupDialog cannot be skipped (project context is required before Home's actions resolve, which the plan asserts at 6.5), say so in R-10 and downgrade the `startupView:'home'` recommendation: "Home is reachable as a tab but cannot be a true zero-gate landing while the directory gate is mandatory; the morning-cue value (1.6, Phase-3 ritual) is weakened by the preceding modal." Do not sell Home-on-open as the morning cue while a modal precedes it every time. This is a decision for Mark (it already half-exists as open question 3), but R-10 currently hides the cost.

## D4 (MEDIUM) — M14d (the notification demotion) never states its DEFAULT, and the plan's own logic points opposite ways; an undecided calm-vs-nag default is the single highest-leverage ADHD knob shipping unspecified

Where: Section 7 / M14d ("a `notifyOnIdle` store flag ... When `notifyOnIdle:false`, the idle toast is suppressed"; the test asserts "default value confirmed" but the prose never says WHAT the default is), Section 9.4 / R-12, open question 10. Cross-ref: R6 item 2 (the round-2 decision) set `onIdle:true` (preserve today's ping). Verified in code: `hook-router.ts:127-131` fires an OS toast on EVERY idle when the tab is not active.

Why it fails the lens: the plan correctly identifies notification amplification as "the biggest activation-energy regression hiding in an activation-lowering plan" (9.4) and "one involuntary switch costs ~23 min re-entry." It then builds M14d to demote the idle toast. But it never decides the default, and the two governing principles collide: R6 item 2's no-regression floor says default `onIdle:true` (do not silently drop the ping Mark relies on); calm-by-default says the dashboard, whose whole purpose is to drive MORE concurrent sessions (9.4), should default the per-session-per-turn toast storm OFF. Shipping M14d with `notifyOnIdle:true` means the calm board the plan just built still fires an OS toast every time any of N sessions finishes a turn, the exact peripheral-interrupt storm 9.4 calls the biggest regression. Shipping it `false` silently removes a ping Mark currently depends on. The plan punts both the decision and the harder question (does the dashboard's existence justify flipping the inherited default?) to "default value confirmed" in a test, which asserts whatever the implementer picked, not whatever is calm.

Minimal fix: decide the default in the plan, do not leave it to the test. The calm-by-default-correct answer, given 9.4's own framing, is `notifyOnIdle:false` ONCE the dashboard is the landing surface (the dashboard IS the idle signal now; the toast is redundant with the board), with `requires_response` toasts preserved (the plan already keeps those). State it in M14d and R-12: "Default `notifyOnIdle:false` because the always-visible Home makes the idle toast redundant and the dashboard amplifies idle toasts across sessions; `requires_response` notifications stay ON because those are a real your-turn signal the board cannot replace at a glance." If Mark wants the floor-preserving `true`, that is open question 10, but the PLAN should carry the calm default and let him override, not ship an unspecified default that a coin-flip in M14d resolves. One sentence + one default value.

## D5 (MEDIUM) — The needs-you list is capped (good), but the "+N more" disclosure has no calm landing: expanding it dumps the full wall the cap was meant to prevent, with no second-level structure

Where: Section 4.6 (the progressive-disclosure cap: "hero + top N rows (N around 4-5), with the remainder behind ONE '+N more' control"), Section 6.3 (the ranked rows capped at hero + N with one "+N more"), Section 6.1 (the needs-you area "scrolls internally on short windows"). This is the round-3 D3 fix, which I credited; round 4 implemented the cap correctly. The new gap is the OTHER side of the disclosure.

Why it partially fails the lens: my round-3 D3 asked for a cap so a 12-15 card heavy week does not present a flat wall. Round 4 added the cap, which fixes the DEFAULT view. But it specified nothing about what happens when the user clicks "+N more": the plan's only stated behavior for the overflow is "scrolls internally" (6.1). So on a heavy week the calm default is hero + 4 rows + "+N more (8)", and clicking it reveals a flat, unstructured, internally-scrolling list of 8 equal-weight rows, the exact "list with no clear starting point" wall (E4 P3/A11) the cap exists to prevent, now one click away instead of zero. Progressive disclosure that reveals an undifferentiated wall on expand is half a fix: it makes the FIRST glance calm and the SECOND glance the original paralysis. For the time-blind brain, "+N more" that opens a wall trains "do not click that," which means the capped-off items silently rot (the AP-F silent-decay pattern the Phase-3 parking spec, PLAN-PHASE-2-3.md, is careful about for todos but the Phase-1 needs-you list is not).

Minimal fix: specify the expanded state, do not leave it as "scrolls." The cheap calm version, reusing structure the plan already has: when "+N more" expands, group the overflow by the SAME age-color bands already computed (4.3) with quiet `text-muted-foreground` mini-headers (the exact pattern 6.4 uses for the session strip's labeled groups), so the expanded list reads as "3 getting old, 5 fresh" not "8 things." No new data, no new mechanism, it reuses `ageColorClass` and the 6.4 mini-header pattern. One render rule, one M8a assertion that the expanded overflow carries group mini-headers (mirroring the M9 strip-group assertion). This makes the second glance calm too, which is the whole point of disclosure.

## D7 (LOW) — The reframe-as-review button label is a fixed canned table (good), but on live data three of the four needs-you decision cards route to the SAME generic "Draft the first version" against a generic slug, so the headline affordance reads as one button repeated, not a smallest-first-move tailored to the task

Where: Section 1.7 (the canned button-label table; `pickPrimaryAction` routes avoidance/decision items to `draftFirstVersion`), Section 3.4 (the canned default is "safe but generic"). Live-verified: the needs-you decision cards (`incomplete-notes`, `od-query-consolidation`, `cad-document-pipeline`, `marketing-roi`) all carry `needs-your-decision` and would route to `draftFirstVersion`, composing "Draft the first version of <slug> so I can review and send it" against slugs like "Od Query Consolidation" and "Cad Document Pipeline."

Why it is a minor lens gap: the reframe-as-review affordance is the plan's named headline ADHD move (1.7, "three recon lanes name this the single highest-payoff affordance"), and the smallest-first-move label discipline is genuinely good. But the plan is honest (1.7/3.4) that the canned default is "safe but generic," and on live data the genericness compounds: most of Mark's needs-you cards are DECISIONS (od-query-consolidation = "earned-only view decision"; cad-document-pipeline = "bespoke-token decision made"), and "Draft the first version of Od Query Consolidation so I can review and send it" is not a coherent reframe of "make the earned-only-view decision." A decision is not a draft. So the headline affordance, on real data, offers to "draft the first version" of things that are not documents to draft, against a slug that does not name the actual choice. The button is safe and the framing is well-intentioned, but a button that proposes drafting a decision reads as the app not understanding the task, which is its own trust erosion.

Minimal fix: this is a known tradeoff the plan already documents, so the ask is small. (1) `pickPrimaryAction` should route `needs-your-decision` items (the producer flags them, `BLOCKER_TAGS` includes `needs-your-decision`) to a DECISION-shaped canned action, not `draftFirstVersion`. The plan already has `reviewTodos` ("Look at the open TODOs") and `summarizeChanges` ("See what changed"); a decision card's smallest first move is "See what's blocking the decision" / "Open the repo to decide," not "Draft the first version." Add one `KnownActionId` for the decision kind (e.g. `openToDecide` -> "Open the repo to decide") and route `needs-your-decision` to it. (2) Or, if no new action, route decision items to `summarizeChanges`/`reviewTodos` rather than `draftFirstVersion`, so the button at least does not propose drafting a decision. Either is one routing line + one `pickPrimaryAction` test asserting a `needs-your-decision` item does NOT map to `draftFirstVersion`. This keeps the zero-free-text PHI guarantee (still canned, still slug-only) while removing the "draft a decision" incoherence on the most common live card kind.
