# Advisor review (round 4): ADHD design coach lens

Reviewer role: ADHD ergonomics advisor. Date: 2026-06-20 (round 4).
Target: `docs/dashboard/PLAN.md` + `docs/dashboard/PLAN-PHASE-2-3.md` (round-4 revision), `dashboard` worktree, HEAD `ce2e9e0` (verified against the appendix anchors).
Lens: deepen the ADHD ergonomics so the design measurably lowers activation energy. Grade against J.O.T. one-thing hero, calm-by-default, sub-2s capture, shame-free avoidance nudges, restrained dopamine feedback.

Hard constraints I checked the plan against and held my own prose to: `claude-terminal-dashboard` worktree only; PowerShell 7 is the shell; no em dashes, no AI-slop words; no PHI in argv/logs/artifacts; AGENTS.md IPC discipline (handler + preload + `global.d.ts` + registration test + explicit remote decision per new channel); every keybinding challenged; vitest test-first for logic; boringly small milestones (one change, one expected test, one rollback point each).

## Verdict

Round 4 closed the round-3 frontier almost completely. The three round-3 must-fixes are in: the button label is now a fixed micro-step table sized to the smallest first move (1.7, the four-row table, "Draft the first version" / "Look at the open TODOs" / "See what changed" / "Open a shell to start"); sub-2s capture survived the move to Phase 2 as a falsifiable synchronous-focus-on-keydown assertion (`PLAN-PHASE-2-3.md` Capture section, M12); "never a goal at zero" is now a global rule with the gap-led frame and a voice-test guard banning "0 of N" on every surface (1.10, 6.6). All six round-3 deepen items also landed: app-open named as the reserved implementation-intention cue (Section 1 intro, 1.9), the per-day parked-id persistence (1.6, M6), the unified glance count that cannot read "0 need you" above a waiting hero (4.6, M8b), the no-streak voice-test guardrail (6.6), pwsh-7-preferred in Phase 1 (3.2, M10), and the one-full-weight-button render assertion (6.3, M8a). On the five lens axes the design is now around **8.7/10**. This is a genuinely calm, J.O.T.-shaped, shame-aware plan, and it is honest about its own limits (R-8, R-13, open questions 12 and 13 name the central bet plainly instead of hiding it).

What keeps it just off 9 is not a missing mechanism and not a regression. It is that the plan has optimized the *read* surface to a very high bar while three high-confidence recon findings about the *act* surface, the seconds between window-focus and first keystroke, are still honored as principle without being pinned to a measurable target or a Phase-1 test. The lens grades on "measurably lower activation energy," and the remaining gap is exactly there:

1. The plan's own success metric (E1 §1: "seconds from window-focus to first keystroke") is never instrumented. Sub-2s capture is pinned as a number; the headline activation path (hero-button-click to a live, query-injected session) is not, even though it is the literal thing the dashboard exists to make cheap.
2. The hero-click-to-ready path has a multi-second dead window (CLI boot, the 30s fail-safe ceiling) with no specified ADHD-calm behavior, so the one moment the plan most needs to feel instant can read as "nothing happened," which for a time-blind avoidance brain is the cue to wander off (E1 AP2, E3 §1).
3. The plan added the goal-gradient pull but did not add the goal-gradient's twin from the same E5 P5 finding: the *finish payout* is a single in-place settle, and the recon's restraint argument is about over-celebration, not about making the one sanctioned reward too quiet to register on a low-baseline-dopamine brain (E5 P6/§5.7, the calibrated-reward point the plan rounds down to "settle + count").

Below: must-fix (in scope for Phase 0/1 as scoped, or a one-line scope decision) and deepen-the-ergonomics. I do not re-litigate what round 4 already fixed; the "kept right" list records it so the fixes do not regress.

## Must-fix (raises the floor toward 9; in scope or a one-line scope decision)

### MF-1. The plan's own success metric, seconds from window-focus to first keystroke, is never instrumented, so the headline activation path is the one ADHD axis with no measurable target

Where: E1 §1 ("The success metric is 'seconds from window-focus to first keystroke in a real session,' not 'information density'"), 3.1 (Open Claude with query, the write-after-ready path with the 30s fail-safe), 1.7 (reframe-as-review, the headline affordance), M10. Contrast: `PLAN-PHASE-2-3.md` Capture, which DOES pin sub-2s as a falsifiable test.

Why it fails the lens at the frontier: the plan correctly made capture's "sub-2s" a number (round-3 MF-2). But capture is object permanence (E1 P5); it is not the activation path. The activation path is the hero button, click to a live session with the query already running, and that path is exactly the metric E1 names as THE success measure for the whole surface. Round 4 pinned the secondary axis and left the primary one as prose. The plan has a 30s timeout (3.1 step 7), which is a correctness ceiling (the query is not silently dropped), but 30s is a failure bound, not a latency target; nothing in the plan says what "fast enough that the avoidance brain does not disengage" is, and nothing tests the proxy for it. A capture test that asserts synchronous focus but a hero-action path with only a 30s outer bound is an inverted priority: the cheaper, less-load-bearing axis is measured and the load-bearing one is not.

This is not asking to measure real wall-clock latency in a unit test (you cannot, and CLI boot is genuinely multi-second and outside the app's control). It is asking for the same move the plan already made for capture: pin the part that IS in the app's control as a falsifiable assertion, so the activation path cannot silently regress.

Fix (Phase 1, small, no new data, no PHI surface):
- In 3.1 / M10, state the activation-latency budget as two SEPARATE numbers Mark signs off, the way the plan states the 150s/10min/45-60s/5-day defaults in one place: (a) the in-app portion, click to "session created, intent armed, tab focused," must be synchronous on the click handler (no `await` before `setActiveTabId` and the intent dispatch to MAIN), asserted exactly like capture's synchronous-focus test; (b) the out-of-app portion (CLI boot to first `idle`) is acknowledged as not-in-app-control, bounded by the 30s fail-safe, and is where MF-2's pending-state UI carries the wait.
- Add to M10's test: clicking the hero primary action dispatches the inject intent to the mocked MAIN channel and calls `setActiveTabId` SYNCHRONOUSLY in the same tick as the click (no `await`, no `setTimeout` before either). This is the testable proxy for "the app's own share of the activation latency is zero," and it is the exact mechanical guarantee behind E1's success metric. It is one assertion in a test M10 already writes.
- One sentence in 1.7 or 3.1: the dashboard's share of "seconds to first keystroke" is asserted to be synchronous; the unavoidable CLI-boot wait is owned by the pending-state surface (MF-2), not left as dead air.

This converts the plan's stated success metric from an unmeasured aspiration into the same kind of falsifiable target the plan already trusts for capture. It is a string in the tunables list plus one synchronous-dispatch assertion, touches no data, widens no PHI surface.

### MF-2. The multi-second hero-click-to-ready window has no specified ADHD-calm pending state, so the one moment the plan needs to feel instant can read as "nothing happened"

Where: 3.1 (write-after-ready: create tab, arm intent in MAIN, gate on first `idle`, 30s timeout), 6.3 (hero anatomy), 6.5 (mandatory states, which cover board loading/empty/error/degraded but NOT the post-click injection window), E1 AP2 (blank-slate launch), E3 §1 (now-vs-not-now: a gap with no signal is "not happening").

Why it fails the lens: the plan is meticulous about the board's first-open timeline never strobing (4.3, the fake-timer test), and it is meticulous that the query is never silently dropped (the MAIN-owned intent + 30s timeout). But between the click and the first `idle`, there is a real multi-second window (CLI boot via `cmd.exe /c claude`, then the idle gate, then the write) where the plan specifies the FAILURE surface (the 30s "Session failed to start" message with retry) but not the SUCCESS-in-progress surface. For an ADHD avoidance brain, the click on a dreaded item is the highest-stakes half-second in the whole app, and E1 AP2 is explicit: landing on an empty prompt "throws the activation cost right back at the user at the worst moment." A freshly created tab that sits blank for three to eight seconds with no "I am working on starting this for you" signal is functionally the blank-slate launch, even though the query is coming, because the time-blind brain reads silence as nothing-happening and disengages before the injection lands (E3 §1, now-vs-not-now). The reframe-as-review affordance (1.7) is the plan's headline move; it deserves a specified in-progress state so its first three seconds do not undo it.

Fix (Phase 1, small, no new IPC channel, no PHI):
- Specify a "starting this for you" pending state on the injected tab (or on the hero) for the window between click and first `idle`: a calm, motion-safe (per 1.2) "Starting your session, the first step will be typed in for you" line, NOT a spinner-per-row (E4 A7) and NOT a countdown (E3 P7). The existing `claude:injectStatus` broadcast (3.1, already in M10) is the signal; this asks the renderer to render its in-progress value, not just its failure value. The state already has the data; the plan just has not said to paint the success-pending case.
- Add to M10's test: while the inject intent is armed and before the simulated first `idle`, the surface shows the pending "starting" affordance (assert the testid/string), and on the simulated `idle` it clears; the existing 30s-timeout test already covers the failure transition, so this is the third arm of a state machine that currently has only two. One render assertion.
- Honor reduced-motion: the pending state is text + at most a motion-safe gentle treatment, asserted to carry no transition class under a `prefers-reduced-motion` mock, mirroring the M8b/M9 pattern the plan already uses. One assertion.

This closes the gap between "the query is never dropped" (a correctness guarantee the plan has) and "the click feels like it did something" (an activation-energy guarantee the plan needs), which are not the same thing. It is a render state the plan's own `injectStatus` channel already feeds.

### MF-3. The one sanctioned dopamine payout is tuned for restraint to the point it may not register; E5 P6/§5.7 ask for a reward CALIBRATED to the achievement, not merely a quiet one

Where: 1.5 (done-lane payoff: "the card settling out with a 150-200ms ease plus the count ticking. No confetti, no streaks"), E5 P3 ("Reward must scale to the achievement"), E5 P6 (low-baseline dopamine: "delayed rewards feel flat; immediate feedback releases dopamine"), E5 §5.7 ("reserve any larger flourish for closing a genuine loop... let the rare big win be the only loud moment"), E5 AP3/AP10 (over-celebration is the trap, NOT under-celebration).

Why it partially fails the lens: the plan reads the dopamine recon as "be restrained" and lands on a single uniform beat (a 150-200ms settle + a count tick) for every finish, the same for clearing a trivial idle tab as for closing a long-avoided financial loop. That is the SAFE direction, and it is right to fear confetti (E5 AP3, E7 C8). But the recon's actual instruction is two-sided: E5 P3 says "tiny check for a tiny task, a bigger rarer beat for a genuine loop closed," and E5 §5.7 says the rare big win should be "the only loud moment," not that there should be no loud moment ever. A low-baseline-dopamine brain (E5 P6, the entire reason this lane exists) needs the finish to actually land; a 200ms ease that is identical whether you cleared a stray shell tab or finally sent the BLOC follow-up trains the brain to not register the payout at all, which is hedonic adaptation arriving from the bland end instead of the loud end (E5 AP3 is over-celebration; the symmetric failure is under-celebration on the one item that mattered). The plan correctly knows WHICH item matters: an avoidance-category card clearing is exactly the "genuine loop closed" E5 P3 reserves the bigger beat for, and Phase 2's classifier (M13) already computes that category.

Fix (stated as a Phase-1 decision, built in Phase 2 with the classifier, so it is not silently dropped):
- In 1.5, replace "one uniform settle" with a TWO-TIER payout that the recon actually asks for: the ordinary finish keeps the calm 150-200ms settle + count tick (Phase 1, as built); a finish that closes an AVOIDANCE-category loop earns one slightly longer, still-calm, still-motion-safe beat (a touch more dwell, never confetti, never sound), reserved so it stays meaningful (E5 P3/§5.7). State it as a decision now even though the avoidance category is Phase 2 data, the same way the plan states other Phase-2-dependent decisions in the Phase-1 doc (Section 8 "decisions a Phase-1 reader still needs").
- Keep the guardrail the plan already has: the bigger beat is still bounded (no confetti, no sound, reduced-motion-gated), and the voice/no-streak tests still apply. This does not reopen the over-celebration door; it calibrates the one reward the plan sanctions so it is felt on the one item it exists to reward.
- One sentence in R-8 or open question 8: the done-lane reward is calibrated by avoidance-category in Phase 2 (closing the loop the app exists to push on is the one finish that earns the rarer beat), so Mark signs off that "restrained" means "calibrated," not "uniform and possibly invisible."

If Mark prefers strict uniformity, that is a legitimate call, but it should be a CHOSEN one, not the default the plan backed into by reading the dopamine recon as one-sided. Flag it; let him pick (this is the symmetric partner to open question 8's streak appetite).

## Deepen the ergonomics (moves 8.7 toward 9)

### D-1. The hero's "not now" escape does not exist in Phase 1, so the first dogfoodable board can pin a dreaded item with no calm exit until Phase 2

Where: 1.1 (the affordance budget: one primary, one quiet "not now" = re-roll, one quiet copy), 1.6 (re-roll is Phase 2, "before the ranker, board order already varies poll-to-poll"), 1.11 (the Phase-1 single-field override), 6.3 (CardFooter: "one quiet 'not now' (the re-roll, Phase 2)").

The plan's reasoning for deferring re-roll is sound for the RANKER (a deterministic ranker is what makes an immovable hero possible, and before it board order varies). But it has a Phase-1 hole the plan half-acknowledges: the single-field override (1.11) is itself deterministic for the two cases it fires on. If a `time_sensitive` card or a `dodAlmost` card is ALSO a dreaded item (a financial deadline, an almost-done SOP), the Phase-1 override will pin it as the hero every poll with no "not now" until Phase 2 ships. On live data this is latent, not active (today's override hero is `practice-reports`, a deploy-watch, not a dread item), so this is a deepen, not a must-fix. But the affordance budget (1.1) lists the "not now" as one of the three sanctioned hero controls, and Phase 1 ships the hero with that slot empty, which means the calmest day-one board can still become a guilt billboard for the exact deadline-shaped avoidance item the override is designed to elevate (E5 5.1, E1 P8, E5 AP6).

Fix (one of two, Mark's call, both small):
- Option A (cheapest, no persistence): in Phase 1 the hero's "not now" is a same-session dismiss that drops the override-selected hero to the producer board head for the rest of the session (no store, no per-day slot, clears on restart). This is honest: it is a session-scoped escape, not the Phase-2 per-day park, and it fills the 1.1 budget slot so the hero is never a no-exit billboard on day one. One render handler, one M8a assertion that dismissing the override hero surfaces the board head.
- Option B (defer cleanly): state in 1.6 and 6.3 that the hero's "not now" slot is INTENTIONALLY empty in Phase 1, and that the override is constrained to never pin the same id as hero more than the producer board already would (i.e. the override only fires when it changes the order, and a dismissed-this-session set is the Phase-2 add). This makes the empty slot a stated decision, not an oversight, consistent with how the plan states every other deferral.

Either is fine; the defect is that the plan currently leaves the slot empty without saying which it chose, so a Phase-1 dreaded-deadline hero has no calm exit and the plan does not acknowledge it. One paragraph resolves it.

### D-2. The body-double / accountability throughline (E7's whole thesis) is gated to Phase 3 default-OFF, and the plan names that risk (R-13) but does not let Phase 1 carry the cheapest possible presence signal

Where: R-13 (the roadmap risk that the differentiated accountability value never ships), 1.9 (commitment mirror, Phase 3), E7 P1 ("the dashboard IS the body double: present, aware, non-chattering"), E7 §1 (the single load-bearing observation: the six avoidance areas rot because they have no external accountability, and external accountability is the one thing Mark responds to and cannot self-generate).

E7 is explicit that the body double works through PASSIVE PRESENCE, not intervention ("their job is to not engage with you," E7 P1/§2). The plan reads "accountability" as "the lock-in ritual" and correctly defers the ritual to Phase 3 default-OFF (the ritual UI is heavy and earns its place last). But E7's actual highest-confidence claim is that presence-and-awareness, not the ritual, is the active ingredient, and presence is nearly free: a Phase-1 board that is always-on, remembers what needs Mark, and is visibly "here" already IS the passive body double E7 P1 describes. The plan ships exactly that and then does not name it, so R-13 reads more pessimistic than the build warrants. The gap is framing, not mechanism: the plan treats "accountability" as a Phase-3 feature when Phase 1 already ships its load-bearing form.

Fix (framing only, no code, raises confidence in the phased bet):
- One sentence in Section 1 (alongside the app-open cue framing): Phase 1's always-on, needs-you-aware Home is the passive body double E7 P1 names (present and aware, not chattering); the Phase-3 ritual adds the active check-in on top of a presence layer that already ships. This reframes R-13 from "the accountability value is entirely gated" to "the passive-presence half ships in Phase 1, only the active-ritual half is gated," which is both more accurate and a stronger argument for why Phase 1 earns daily opens (the gate's precondition).
- No mechanism change. It is the framing that tells the M17 builder the presence layer already exists and the ritual is the increment, exactly as round-3 D-1 did for the app-open cue.

Where: 5.2 (the idle-age floor promotes a waiting tab into Tier 2), 4.6 (empty-board live-tab hero eligibility), 6.4 (the strip's `requires_response` row), 6.6 (copy voice rules), E7 P7 ("treat 'Claude is waiting on you' as a real accountability item"), E7 C9 ("letting completing-the-loop items die silently... is the failure being solved, not a calm default"), E7 F5 (the copy: "Your cad-portal session has been waiting ~10 min").

What it never specifies is the ACCOUNTABILITY COPY E7 P7/F5 ask for: the waiting-loop item, when it crosses the floor and especially when it becomes the hero, should read as a present partner noticing ("This session has been waiting on you 6m"), not as a status label ("idle 6m"). The difference is small in characters and large in register: one is a body double, one is a clock. The strip row can stay terse ("idle 6m") since it is subordinate; the PROMOTED form is where the accountability register matters.
- Add to the voice test: the waiting-loop hero/promotion string is present in the module, em-dash-free, slop-free, and contains no guilt framing ("still", "again", "keep") per E7 C2/C3. One assertion, in the test M8b/M9 already extend.
- One sentence in 5.2 or 6.4: the waiting-loop item, when promoted, reads as present-partner accountability (E7 P7), not a status timestamp; this is the chime Mark used to listen for, turned into one quiet, specific, shame-free line.

This is the single place the plan's mechanical excellence (the `waitingSince` spine) is not matched by its copy, and it is the exact item E7 says the app exists for.

### D-4. The "Pull one forward?" caught-up affordance hands Mark a fresh choice at the one moment the plan should be deciding for him

Where: 4.6 (the caught-up state: "one quiet, dismissible 'Pull one forward?' surfacing the single calmest active card"), 6.5 (mandatory states), E4 F9 (empty/done reads as momentum, "want to pull one forward?"), E1 P2 / E5 AP7 / E7 C7 (the one-recommendation rule, decide for him, never a menu), user profile ("don't ask 'what would you like to do?', say 'here's what I'd do, want me to proceed?'").

The caught-up pull-forward is a good instinct (E4 F9, don't let the cleared board dead-end). But the plan phrases it as a question, "Pull one forward?", which is the one phrasing Mark's own profile and three recon lanes say to avoid: at the moment of a cleared board, the brain is at its lowest activation, and a question ("which one? do you want to?") reimposes the exact choice the J.O.T. hero exists to remove (E1 P2, E7 C7). The plan already DECIDED which card it surfaces ("the single calmest active card"), so it is doing the work of picking; it just presents the result as a question instead of a recommendation. This is the difference between a coach ("here's a calm next one, want it?") and a menu prompt ("pull one forward?").

Fix (Phase 1, one copy string, no new logic, the card is already chosen):
- Reword the caught-up pull-forward from a question to a one-recommendation offer, in the pinned copy module: "Next calm one: [card], want it?" or "When you're ready: [card]." The card is already selected by the plan's own "single calmest active card" rule; only the phrasing changes from a choice to a recommendation. One string, covered by the existing voice test (which already asserts no slop / no em dash; add that the caught-up offer names the chosen card, not an open "which?").
- One sentence in 4.6: the pull-forward is a single recommendation (the chosen calm card), not an open question, per the one-recommendation rule (E1 P2, user profile). It stays one quiet dismissible line.

This is a copy change, not architecture, and it aligns the one caught-up affordance with the rule the rest of the plan obeys everywhere else.

### D-5. The minute-coarsened count-up is correct, but the idle-age FLOOR has no ambient pre-floor cue, so a tab that is about to need Mark is invisible until it crosses, which is a now-vs-not-now cliff

Where: 5.2 (the idle-age floor: a tab waiting less than ~45-60s stays in the subordinate strip, only after crossing does it enter the needs-you count/list), 1.2 (minute-coarsened count-up), E3 §1 (now-vs-not-now binary), E3 P4 (make not-now visible on a duration ramp), E3 §4.7 (color drift along the ramp BEFORE bubbling to the hero).

The floor is the right mechanism (it stops badge-soup from per-turn idle fires, E4 A1/A3). But the plan implements it as a hard threshold: below the floor a tab is fully subordinate, at the floor it jumps into the needs-you band. E3's whole point is that the time-blind brain experiences exactly that kind of step as now-vs-not-now with nothing in between, and E3 §4.7 prescribes the antidote: a duration ramp where the item drifts in salience as it ages, so crossing is a continuation, not a surprise. The plan has the ramp data (the strip already shows the waiting duration) but the strip row is uniformly subordinate until the floor, so there is no ambient "this one is getting close" cue. The result is that a tab Mark could pre-empt is fully quiet, then suddenly in his needs-you count, which is the cliff E3 names.

Fix (deepen, small, Phase 1 strip-only, no new data):
- In 6.4, let the strip row carry a SUBTLE pre-floor age treatment (the existing low-S age-color drift on the duration string, already in the plan's vocabulary via `ageColorClass`) so a waiting tab warms slightly as it approaches the floor, then crosses into the needs-you band as a continuation rather than a jump. This is the E3 §4.7 ramp applied at session scale, and it reuses the age-color machinery the plan already built (4.3, M7).
