# Adversary review: Integration / feasibility skeptic

Lens: attack any mechanism that may not actually work on the `dashboard` branch. Write-after-ready timing, `fs.watch` on atomic rename, hero-ranking under the unreliable `requires_response`, remote parity, the renderer-only Home guards.

Target: `docs/dashboard/PLAN.md` (round-3 revision), recon `docs/dashboard/recon/round2/R1-R6`.
Build target verified: worktree `infrastructure/claude-terminal-dashboard`, branch `dashboard`. Every code citation below re-resolved against this checkout, and the live `state.json` was read at `generated_at 2026-06-20T21:12:42`.

## Verdict up front

The plan is unusually well-grounded. I tried to break the load-bearing mechanisms and most held: the render seam line numbers (`App.tsx:576-583`), the flat `updateStatus` setter (`tab-manager.ts:55-58`), the `tab:ready -> idle` chain (`hook-router.ts:98-107`), the atomic-write producer (`state.py:6-11`), the 5-day needs-you window and the no-`>=2`-guard `dodAlmost` predicate (`status.py:61,66`), the 6-char/31-alphabet token (`web-remote-server.ts:57-58`), the remote `tab:create` cwd discard (`:322-323`), and the `--plan` bug (`types.ts:72`) all check out exactly as the verification log (PLAN.md:799-819) claims. The phase re-cut is sound and the security milestones are pulled to Phase 1 where they belong.

What survives my attack is a set of TIMING and DATA-REALITY gaps that ship green and fail (or no-op) in-app. The two that genuinely bite are the renderer-reload orphan of write-after-ready (the fail-safe dies with the thing it is supposed to fail-safe) and the done-lane carrot being mostly unreachable on the real board (11 of 18 programs structurally cannot pay it out). The rest are smaller but real.

## Defects

### D1 (HIGH) — Write-after-ready and its mandatory 30s fail-safe both die on a renderer reload, silently dropping the injection

PLAN.md 3.1 and R3's sketch put the entire mechanism in renderer state: `pendingInjection` is a `useRef<Map>` in `App.tsx`, the once-flag lives in that ref, and the M10 "MANDATORY 30s timeout" (3.1 step 7) is a renderer-side `setTimeout`. The plan sells the timeout as the fail-safe for exactly the ADHD case: "clicked 'decide this' and walked away" (3.1 step 7).

The app has a first-class renderer-reload recovery path I verified at `App.tsx:302-316`: on mount it calls `getTabs()`, and if tabs already exist in main it rehydrates `tabs`/`activeTabId` and returns. A renderer reload (Ctrl+R, a Vite HMR hiccup in dev, an Electron renderer crash-and-recover, or the user reloading) between `createTab` and the first `idle` wipes the ref Map AND cancels the 30s timer. The spawned Claude tab still exists in main and reaches `idle`, but nothing is listening for it, so the query is never written and the fail-safe never fires. The user walked away expecting either the query to run or a "failed to start" surface; they get neither. This is the precise silent-drop the plan calls "the worst failure" (3.1 step 7), reintroduced through a path the plan never considers.

It is also not purely theoretical: write-after-ready can take many seconds (Claude CLI boot + SessionStart), and the reload window overlaps it.

Minimal fix: persist the pending-injection intent in MAIN, not the renderer, OR at minimum re-arm on rehydrate. Cheapest: when `createTab` is called for an injection, stash `{tabId, query, deadline}` in a main-side map and have main perform the write-after-`idle` (main already sees the `idle` transition in `hook-router.ts:104` before it broadcasts). That also makes the timeout survivable. If keeping it renderer-only is required, the rehydrate path at `App.tsx:306-316` must reload any persisted pending intents and re-attach the once-flag + timer; document that a reload before persistence is a known unrecoverable gap. Add a test that drops the ref mid-flight and asserts the query is not silently lost.

### D2 (HIGH) — The done-lane "N closed since open" carrot is unreachable for the majority of the real board

PLAN.md 1.5 / 4.6 / 6.3 make "N closed since open" the single Phase-1 carrot, "scoped HONESTLY to program-card Done-lane crossings only." The producer puts a card in `lane:done` only when `dod.total > 0 && dod.met == dod.total` (`status.py:32-34`). I read the live board: 18 programs, and only 7 have `dod.total > 0` at all. The other 11 are `lane:active` with `dod.total: 0` (`lane` distribution live: `{active:11, blocked:5, paused:1, done:1}`). A program with no DoD can NEVER cross into `done`; finishing its work just leaves it `active` or ages it to `backlog`. So for 61% of the tracked programs, completing them produces zero payoff pixels.

This is not a correctness bug (the count is honest), but it guts the carrot's purpose for a low-dopamine brain: the most common "I finished something" event on this user's actual board (commit/merge a program that has no DoD checklist) lights up nothing. The plan's own thesis (1.5: "the finish must pay out") is contradicted by the data the feed actually carries. The plan acknowledges the count is snapshot-derived and suppressed-when-zero, but never acknowledges that the dominant completion event is invisible to it.

Minimal fix: state this limitation explicitly in 1.5 and R-8 (the carrot fires only for the ~7 DoD-bearing programs; no-DoD program completion is invisible in Phase 1). Then either (a) broaden the resolved-set to also count a card LEAVING the needs-you set across polls (the plan already tracks needs-you membership for `justResolved`, 1.5 — a needs-you card clearing IS a felt finish and covers blocked/active cards), or (b) defer the carrot honestly rather than ship a payoff that the board mostly cannot trigger. Option (a) is cheap and uses data already in hand.

### D3 (MEDIUM) — Tier-2 "longer-idle first" tie-break reads `statusSince`, but the transition guard makes `statusSince` the WRONG clock for idle duration

PLAN.md 5.3 Tier 2 and 5.4 sort within-tier by "longer-idle (larger `now - statusSince`) first," and 6.4 computes the idle-duration string ("idle 6m") off `statusSince`. But the new `updateStatus` (2.2 snippet) stamps `statusSince` only on a status CHANGE. R2/R3 establish that `idle` fires once per turn and lands repeatedly (Stop hook every turn, plus `tab:ready` re-fires). A tab that goes `working -> idle`, then the user does nothing, stays `idle`; `statusSince` correctly marks the idle entry. Good so far. But the same tab on its NEXT turn goes `idle -> working -> idle`: each `idle` after a `working` is a real transition, so `statusSince` resets. That is correct for "how long idle THIS time."

The defect is subtler: the plan's idle-age FLOOR (5.2) and the Tier-2 "longer-idle" ordering both assume `statusSince` measures "how long since Claude last did anything for me." It does not when `requires_response` is layered in. If a tab goes `idle` (statusSince=T0), then a Notification fires `requires_response` (statusSince=T1), the overlay is set but the idle clock is now reset to T1 even though the human has been ignoring it since T0. The escalation heuristic (older = more urgent) under-ages exactly the tab that has been waiting longest with an unanswered notification. The plan treats `requires_response` purely as an additive boost (1.x, R2) but never reconciles that the boost transition mutates the very timestamp the floor and tie-break depend on.

Minimal fix: separate the clocks. Keep `statusSince` for display, but drive the idle-floor and the longer-idle tie-break off `lastActivityAt` is also wrong (it updates every call). Introduce a dedicated "waiting since" anchor that is set when the tab first becomes human-waiting (`idle` with `firstActivityAt != null`, OR `requires_response`) and is NOT reset by an `idle -> requires_response` transition within a continuous waiting span. Cover it with the M9/M1 timer tests. Without this, the "N need you" floor and the urgency order are quietly inconsistent the moment a Notification lands on an already-idle tab.

### D4 (MEDIUM) — `fs.watch` re-arm on a deaf/closed handle: the plan promises re-arming but the failure mode is "fires once then silent," which throws no error to re-arm on

PLAN.md 4.3 item 2 and R-2 say the directory watcher "is re-armed on error and the ~20s poll is the tested backstop," and M4 tests "the watcher never arms or throws." That covers the throw path. It does not cover the documented Windows failure the recon itself names (R4 §(c)): `fs.watch` on Windows "can go deaf after one event and does not auto-recover." Going deaf is NOT an error event; the watcher object stays alive, emits nothing, and throws nothing. There is no signal to "re-arm on error" because there is no error. So the re-arm mitigation does not actually address the failure it cites; only the ~20s poll does.

This is fine in practice (the poll is the real load-bearing mechanism and the plan says so), but the plan over-claims the watcher's robustness and M4's test surface implies a re-arm path that the dominant deaf-failure never exercises. Worse, a builder reading "re-armed on error" may add a watcher they believe self-heals and lean on it for snappier-than-20s updates, then ship a board that silently lags by up to 20s whenever the watch goes deaf, with no telemetry that it happened.

Minimal fix: drop the "re-armed on error" language as a robustness claim. State plainly: the watcher is best-effort latency reduction only, the deaf-handle failure is undetectable, and correctness rests entirely on the poll. If snappier updates matter, periodically tear down and recreate the watcher on a timer (e.g. every poll tick) rather than relying on an error that never comes. Adjust the M4 DoD to test the poll-only correctness path and NOT to imply the watcher self-heals.

### D5 (MEDIUM) — Cold-start ordering: the hero's primary action is unavailable in exactly the window the user first looks at the board

PLAN.md 3.1 step 2 requires `activeProjectIdRef.current` to be set, else the Open-Claude action surfaces "Open a project first" and stops. PLAN.md 6.5 + 1.11 make Home the optional default landing surface (`startupView:'home'`, M14). Trace the startup machine: when `startupView:'home'`, the plan routes `setActiveTabId(homeId)` at the three startup sites (M14b). But `activeProjectId` is set inside the startup effect (`App.tsx:300` `setActiveProjectId(projectId)`) which runs only after a directory/project is resolved. If Home is the landing surface and no project has been opened yet (first run, or a fresh launch where the user lands on Home before picking a folder), the hero renders with its primary action permanently showing "Open a project first." The headline affordance (1.7, "the single highest-payoff ADHD affordance") is dead on the default surface for the first-run user, who is precisely the one the dashboard is trying to activate.

The plan never sequences "Home is the landing" against "the actions need an active project." It is internally consistent only if a project is always already active when Home shows, which is not guaranteed by the startup paths I read.

Minimal fix: define the no-active-project Home state as a first-class case (6.5 lists empty/loading/error/degraded but not "no project yet"). Either auto-resolve a default project for Home actions (the workspace root, which `tab:create` already falls back to at `ipc-handlers.ts:357-360`), or make the hero action route through a "pick where this runs" step instead of a dead "open a project first." Add a test for the Home-landing-with-no-active-project case.

### D6 (LOW) — Remote `tab:rename` unbounded-trust is cited as the bad precedent, but the SAME unbounded `msg.name` is what flows into the broadcast `tab.name` the plan elsewhere treats as sensitive

PLAN.md 8.5 correctly flags `web-remote-server.ts:297-307` as the "trusts `msg.name` with no bound" anti-pattern not to copy for capture. But R-11 (PLAN.md:780) separately notes `tab.name` is auto-summarized work content broadcast to synced clients. These are the same field: a remote client can `tab:rename` arbitrary unbounded text into `tab.name`, which then rides `tab:updated`/`tabs:sync` to every other synced client and (per M0b's scope) must be kept out of logs. The plan treats the rename-trust as a Phase-2 capture concern and the name-broadcast as a Phase-1 residual, but does not connect that the rename channel is a live write primitive into the sensitive broadcast field TODAY, before any dashboard code. C1 (token hardening) is the only guard, which the plan does say for `pty:write` (R-11) but not for `tab:rename`.

Minimal fix: fold `tab:rename`'s unbounded `msg.name` into R-11's "the token is the only guard" register, and note it as a pre-existing write-to-broadcast path (not a dashboard regression, but in the same blast radius C1 protects). Optionally bound `msg.name` length in the same C1 commit since it is one line next to the token change.

### D7 (LOW) — The Tier-1 hero override and the live board both make the override fire, but the plan's "1.11 fixes newest-commit" justification is weaker than stated because the residual order is ALSO newest-commit

PLAN.md 1.11 justifies the single-field override as rescuing the hero from "whichever needs-you card has the newest commit." On the live board all 6 needs-you cards are `age_days` 0 or 2 (5 at 0, one at 2), so the producer's `git.age_days`-ascending order (`poller.py:115`) is nearly a tie and the "newest commit" hero is essentially arbitrary among same-day cards. The override correctly elevates `practice-reports` (time-sensitive, 2 days out) to hero — verified live. So the override works. But the plan oversells the problem it solves: with everything committed today, "newest commit" is not a meaningfully wrong hero, it is an arbitrary one, and after the override fires for the deadline/almost-done cases, the REST of the list (PLAN.md 1.11 "the rest stays in board order") is still newest-commit-first, i.e. the same ordering the override is framed as correcting. The thesis "one most-important thing" is satisfied only at slot 0; slots 1..n carry the contradiction the plan says it removed.

Minimal fix: cosmetic but worth honesty. Reframe 1.11 as "elevate a deadline/almost-done card to the hero slot; the remaining list intentionally mirrors the producer board the user also sees" (which is the real, defensible design from 5.4's consistency argument), rather than claiming it fixes a wrong order. No code change; it prevents a Phase-2 builder from "fixing" the residual list order and diverging from the producer board.

### D8 (LOW) — M0b logger gate decides the pending-buffer behavior, but the pending buffer exists to replay logs to DevTools once it opens; gating debug/info out of it changes diagnostic behavior the plan calls benign

PLAN.md 3.6 / M0b gate the `executeJavaScript` mirror to warn/error and decide that "gated-out levels skip BOTH the live mirror and the pending buffer (the `else` at `:48` is inside the gate)." The verification log (806) confirms the pending buffer is the `else` branch when the renderer is not ready. The plan treats this as a clean security win. The behavioral cost it does not name: today, debug/info emitted BEFORE DevTools is attached are buffered and replayed when it opens; after M0b, they are dropped from the renderer console entirely (still on disk). That is the intended security outcome, but it silently changes the dev experience (early-startup debug logs vanish from DevTools), which a future developer will hit and "fix" by re-opening the gate, reintroducing the leak. The plan should pin this as a deliberate behavior change with a comment at the gate, not just a security note.

Minimal fix: add a one-line code comment at the gate explaining that early debug/info are intentionally disk-only and must not be re-enabled in the mirror, and note the dev-experience change in M0b's DoD so it is not later reverted as a regression. Disk still has them for diagnostics (the plan's point), so nothing is lost.

## Things I attacked that HELD (so the plan owner knows they were checked)

- The renderer create-then-subscribe race (R3 sketch): `pendingInjection.set` runs synchronously after `await createTab` resolves; the `idle` event is hook-driven and arrives hundreds of ms later via the pipe, so the ref is always populated first. The intermediate `tab:updated` with `status:'new'` (fired synchronously inside `wirePtyToTab` at `ipc-handlers.ts:109`) is harmless because the gate is on `'idle'`. Feasible. (Worth a test, but not a defect.)
- CRLF handling: `query.replace(/\r?\n/g, '\r') + '\r'` matches xterm's Enter byte (`Terminal.tsx:148-150`) and the existing `\x0c` precedent. Correct.
- `requires_response` not the spine: R2's idle+hadActivity replacement is sound and the `firstActivityAt` field survives reload and rides `tabs:sync`. The self-healing idle-first edge (2.2) is real and correctly pinned. (The clock-reset interaction is D3, separate.)
- Home-out-of-TabManager: verified the web-client render map iterates `tabs` only (`main.tsx:386-394`) and never imports `HomeView`/`App`; keeping Home renderer-only genuinely closes the phantom-tab and activation-count hazards. The separate-slot + self-enforcing `onTabUpdate` guard (2.3) is belt-and-suspenders but correct.
- Remote parity matrix (3.5): the cwd discard (`web-remote-server.ts:322-323`), the `createShellTab` throw (`ws-bridge.ts:261-263`), and the clipboard-over-HTTPS path all match the code. Desktop-only Phase-1 Home makes the disabled-state code latent but correctly staged.
- Age-color verbatim consumption and the off-by-one bands (`status.py:22-29`): exact.
- `generated_at` naive-local vs offset-aware fields: both forms present in the live file; the two-parser design removes the misuse surface as claimed.
