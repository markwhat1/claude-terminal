## Must-fix (would ship green and fail in-app, or violate a hard constraint)

### MF-1. M5 broadcast helper edit is not on the milestone's own change line, and the remote-forward test wiring is underspecified for the "spy" path

Section 2.4 and the 2.10 table say the `program-board:state` push is added inside `sendToRenderer` (`index.ts:75-99`, forward chain `:80-98`), and M5 offers two options: (a) extract `REMOTE_FORWARDED_CHANNELS` and assert absence, or (b) spy `webRemoteServer.broadcast`. Option (b) is the lower-blast-radius choice, but the appendix (`:805`) verified the forward set is an inline if/else-if that calls `webRemoteServer.broadcast(...)` directly. Spying that requires the test to import the real `index.ts` module and inject a `webRemoteServer` test double, which today has no seam: `sendToRenderer` is module-private and reads a module-scope `webRemoteServer`. The plan asserts the test is buildable but does not name the injection seam.

Decision the plan must make in M5's DoD: either (a) is actually cheaper here because it introduces a pure exported constant that is trivially unit-testable with zero module-harness work, OR (b) requires a small refactor to make `webRemoteServer` injectable (e.g. the existing `deps` object the hook router already uses). Pick (a) unless a `deps`-style seam already exists for `sendToRenderer`. As written, "pick one of (a)/(b)" leaves the hardest part (the test harness for (b)) unscoped, which is the same "unbuildable assertion" trap the plan correctly killed for the registration test. Name the seam or commit to (a).

### MF-2. The `onProgramBoardState` preload listener has no registration assertion, only a `typeof api` type-presence check

AGENTS.md requires every new channel to have an assertion in the registration test. M5 adds TWO renderer-facing surfaces: `program-board:getState` (invoke, covered by the `expectedHandlers` membership check) and `program-board:state` (the `ipcRenderer.on` broadcast consumed via preload `onProgramBoardState`). The plan tests the broadcast's REMOTE decision behaviorally, and tests `getState` registration, but the `onProgramBoardState` listener wiring itself (that the preload actually registers an `ipcRenderer.on('program-board:state', ...)` and the renderer receives it) is only covered by a `typeof api` compile check. A `typeof api` check proves the METHOD EXISTS on the bridge; it does not prove the channel name matches between `sendToRenderer('program-board:state', ...)` and `ipcRenderer.on('program-board:state', ...)`. A one-character channel-name typo ships green. Add to M5: a test that drives a `program-board:state` send and asserts the registered preload callback fires (the existing test harness already captures `ipcRenderer.on` registrations the same way it captures `handle`/`on` for the handlers map). This is the difference between "the bridge has a method" and "the live push reaches the renderer."

### MF-3. The injected query cwd-mismatch hazard for an existing-project tab is not fully closed for the LOCAL path

3.1 step 2 resolves `projectId` via `activeProjectIdRef.current` and stops if none is active. But the hero is a PROGRAM-BOARD card whose repo (`program.repos[0]`) is frequently NOT the active project. The PowerShell action (3.2) correctly resolves `WORKSPACE_ROOT + program.repos[0]` as cwd. The Open-Claude-with-query action does NOT: it creates a tab in `activeProjectIdRef.current`, so the canned query runs against whatever project happens to be active, not the program the user clicked. This is the SAME class of defect the plan flagged and disabled for the remote path (wrong-tree action), present silently on the LOCAL path. Either (a) resolve the program's project (find-or-add the project for `program.repos[0]`, then `createTab(thatProjectId, ...)`) before injecting, or (b) state explicitly that the Phase-1 local action runs in the active project and is therefore only offered when the hero's repo matches the active project (and disabled otherwise with helper text, mirroring 3.2). Option (a) is the right answer for an ADHD "decide this now" affordance; (b) is acceptable if documented. As written, 3.1 is internally inconsistent with 3.2 on the exact cwd-correctness point 3.6 treats as security-relevant.

### MF-4. There is no `orange` age-color token; M7's `ageColorClass` must map four producer colors onto four existing tokens, and the plan never states the map (verified)

I grepped `globals.css`. The defined status tokens are exactly four: `--success: #6a9955` (green), `--warning: #dcdcaa` (yellow), `--attention: #ce9178` (orange-ish), `--destructive: #f44747` (red). There is NO `--orange` and no red/yellow/green/orange "age color" token set. But the producer emits a FOUR-value `age_color: green|yellow|orange|red` (4.3, `status.py:22-29`), and M7's `ageColorClass` is specified only as "maps the verbatim colors to token classes." The mapping is unstated, and the obvious one collides: green->success, yellow->warning, orange->attention, red->destructive. That reuses `--attention` (the orange) for BOTH the age-orange band AND the hero primary accent / single highest needs-you dot (1.2 reserves `--attention` as "the one saturated accent"). So a hero whose age band is orange paints the band in the same token as its primary button, defeating the "one accent" calm-by-default rule for a very common case (live data has 7-13 day cards). M7's DoD must (a) state the four-to-four mapping explicitly and (b) resolve the `--attention` double-duty: either give the age-orange band its own muted class (not the saturated accent) or accept the collision and document it. As written, M7 asserts "age-color mapping to token classes" against a token set that has no orange and a collision on the one it would reuse. This is a correctness bug in a function that HAS a test, caught now instead of at paint.

(`TabIndicator.tsx:15` is `text-warning animate-spin`, `:27` is `text-attention animate-pulse`, confirming both the M2 bare-animation targets and that `requires_response` already uses `text-attention` for the glyph, a third consumer of the hero accent token. The strip reusing TabIndicator (6.4) therefore puts `--attention` on every `requires_response` row, so the hero is not the only place the accent appears; 1.2's "reserved for the hero" claim is already false the moment the strip renders. State the accent budget as "hero primary + requires_response glyph + highest needs-you dot," not "hero only.")

## Should-fix (raises quality / closes a real gap, not a green-and-broken ship)

### SF-1. M0b's "skip the pending buffer for gated-out levels" needs the early-window ordering pinned

3.6 and M0b decide gated-out (debug/info) levels skip BOTH the live mirror and the pending buffer because the `else` at `logger.ts:48` is inside the gate. Correct for steady state. But the pending buffer exists precisely for the window BEFORE the renderer/DevTools is ready (`emit` runs, no window yet, push to pending, flush later). After M0b, a warn/error emitted during that early window must STILL buffer-and-flush, while debug/info must not. The M0b test as written spies `executeJavaScript` (proves warn/error reach the mirror eventually) but does not assert the EARLY-WINDOW warn/error path still buffers and flushes once the window arrives. Add that case, or you risk a gate that accidentally drops early-boot warnings/errors (the ones you most want) along with the debug noise. One extra assertion.

### SF-2. The idle-age floor (5.2) interacts with `firstActivityAt` resetting semantics across `/clear`

`idleNeedsYou = idle && hadActivity && idle-age >= floor`, with `hadActivity = firstActivityAt !== null`, stamped once on first `working` (2.2). But `/clear` re-emits `tab:ready` -> `updateStatus(idle)` (R3 §(b), hook-router `:82-96`) and resets the tab name. After a `/clear`, the tab has `firstActivityAt` already set (from its pre-clear working) and lands on `idle`. So a freshly-cleared tab the user just reset is immediately `idleNeedsYou`-eligible the moment it crosses the floor, surfacing as "needs you" when the user deliberately cleared it to start fresh. This is probably fine (a cleared idle tab IS waiting for input), but it is an UNSTATED behavior on the R2 spine, and the M1 edge-test pins only the idle-FIRST case (`firstActivityAt` stays null). Add a one-line decision: does `/clear` reset `firstActivityAt` to null (treat as fresh-ready) or keep it (treat as waiting)? The plan stamps `firstActivityAt` once and never clears it, so the implicit answer is "keep it / treat as waiting." State that, and add it to M1's edge coverage, so a reviewer does not "fix" it either direction.

### SF-3. `statusSince` for the count-up has a cold-start null on tabs that predate the field

2.2 adds `statusSince`/`lastActivityAt`/`firstActivityAt` init to `null` at creation. The working count-up (1.2, 6.4) computes `now - statusSince`. On the very first launch after the field is added, tabs RESTORED from session persistence (`index.ts:111` filters `type==='claude'` for persistence; restored tabs are re-created via `createTab`) will have `statusSince: null` until their next status change. A `working` tab restored with `statusSince: null` would compute `now - null = now` (a giant elapsed time) or throw, depending on the `formatRelative` guard. M7's `formatRelative` test must include the `statusSince === null` case (render "working" with no time, or "just now", never `NaN`/`Infinity`/a 56-year count). The plan says `formatRelative` is TDD'd but lists only boundary cases, not the null-anchor case that restored tabs guarantee on first run. Add it.

### SF-4. M3a's extracted-helper test does not cover the render seam it claims to prove

M3a's change includes "the sibling render seam after `tabs.map`" but its test is the extracted pure helpers `selectActiveView` and `computeTabCounts`. Pure-helper tests do not prove the JSX seam (that `{activeTabId === HOME_TAB_ID && <HomeView/>}` actually renders next to the map and that xterm is never instantiated for Home). M0 establishes the App.tsx mount harness specifically because no test mounts App today. M3a should USE it: a render assertion that with `activeTabId === HOME_TAB_ID` the HomeView placeholder is in the DOM and no `Terminal` for a home id mounts. Otherwise the seam (the thing most likely to be mis-placed inside vs outside `data-terminal-area`) is untested while the trivial helpers are over-tested. This is the one milestone where the DoD's "one structural observable" should be a DOM observable, not a pure-function return.

### SF-5. `consolidateAttention` red/yellow/green vs the four-band age color (6.4 / M7) is a two-vocabulary mismatch

6.4 says the aggregate nav badge collapses "red > yellow > green" but the producer's `age_color` is a FOUR-value band (green/yellow/orange/red, 4.3). `consolidateAttention` in M7 must collapse FOUR colors (orange exists and is common: 7-13 days), not three. The plan's prose drops orange in two places (6.4 "red > yellow > green", 5.4 tie-break "red, orange, yellow, green" gets it right). Reconcile: the consolidation order and the tie-break order must be the same four-element ordering. M7's test should assert orange is ranked between red and yellow, or an orange-only board shows no nav badge (a real Phase-1 case given the live data ages). Small, but it is a correctness bug in a function with a test, which is exactly where it should be caught.

### SF-6. Capture id minting cites `tab-manager.ts:4-6` but that helper is module-private

8.2 mints capture ids "like tab ids (`tab-manager.ts:4-6` pattern)". I verified `generateId()` at `tab-manager.ts:4-6` is a module-private function, not exported. M12 either duplicates the `tab-${Date.now()}-${Math.random()...}` literal (fine, but say so) or the plan should note it is a PATTERN to mirror, not a symbol to import. As written a builder may try to import it and find it unexported. One word ("mirror the pattern") removes the ambiguity. Better: since both tab ids and todo ids want the same shape, consider exporting `generateId` from a shared util in `src/shared/` and reusing it (maximize-reuse lens), which also makes it unit-testable for collision-resistance. Minor, but it is a reuse opportunity the plan's own `src/shared/` consolidation principle (2.7) argues for.

## What the plan gets right (worth preserving under edit pressure)

- Home renderer-only with a SEPARATE state slot (not in `tabs`), plus the self-enforcing `onTabUpdate` guard (`if (tab.type === 'home') return prev;`) so the invariant does not depend on six disciplined call sites. This is the single best architectural decision in the plan and directly answers R1's "count real tabs only" with a structural guarantee instead of an audit.
- The third additive Tab field reasoning, and the transition-guarded `updateStatus` (stamp `statusSince` only on change) catching the per-turn idle re-fire that would otherwise corrupt the Tier-2 tie-break. That is a subtle, correct read of the R2/R3 idle semantics.
- Two named timezone parsers instead of one flagged function (2.8). Removing a misuse surface by name+compiler rather than relabeling it as a flag is exactly right and testable.
- The full AGENTS.md five-part treatment on the one new local channel, with the remote decision tested BEHAVIORALLY (not by membership in a list that does not exist). The plan correctly discovered the forward set is an inline chain, not exported data.
- Consuming the producer's `dod`/`age_color`/`time_sensitive`/`needs_you` VERBATIM and the catch that the producer's almost-done predicate has no `>= 2` guard (so a single-item DoD IS hero-eligible). That is the kind of contract-fidelity bug only a real read of `status.py` finds.
- Security folded into Phase 1 (C1 token width + connection-attempt bound, M0b logger gate) rather than a "confirm later" check, with honest keyspace math (31^6 = 8.87e8) and the honest admission that the injection guarantee is LOCAL-ONLY because raw remote `pty:write` bypasses the choke point (R-11). Naming the residual instead of overclaiming the control is the correct posture.
- Milestone discipline: phase re-cut so each ships before the next is built, DoDs that are green-test-plus-structural-observable, and the `src/shared/` consolidation that serves both shells from one source and asserts web-client vite-bundle resolution, not just vitest.
